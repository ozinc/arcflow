# ArcFlow Licensing — Plain-English FAQ

**ArcFlow is free to build with, free to ship within generous limits, and paid when you scale or need premium capability.** This file answers the questions developers and AI coding agents ask most.

For the legal text, see [`LICENSE`](./LICENSE). For business pricing, see [oz.com/pricing](https://staging.oz.com/pricing). For an AI-agent-specific summary, see [`ARCFLOW_FOR_AI_AGENTS.md`](./ARCFLOW_FOR_AI_AGENTS.md).

## The 30-second version

> ArcFlow Core is a free-to-use proprietary engine. The SDKs, cookbook, examples, install scripts, and protocol spec in this repo are MIT-licensed. You can build, test, ship, and run commercial applications with ArcFlow without paying. Paid plans exist for managed cloud, distributed clusters, premium algorithms, enterprise governance, SSO, audit logs, SLA, and custom deployment rights.

Reference class: GitHub, VS Code, Docker Desktop, Copilot, Cursor, Vercel, Cloudflare Workers, Firebase, Hugging Face, Unreal Engine. All proprietary or source-available cores with permissive developer ecosystems. ArcFlow follows the same model — closest single analog is **NVIDIA CUDA** (closed runtime, free developer surface, open documentation).

## Can I use ArcFlow commercially?

**Yes**, within the Free Use Limits documented on the [pricing page](https://staging.oz.com/pricing). You can build, ship, and run commercial applications — startups, internal tools, SaaS products, customer-facing services — without paying for ArcFlow Core.

We deliberately do not gate basic commercial use behind a paywall. The reference is Firebase Spark plan and Unreal Engine's revenue threshold: "free even in production, within documented limits."

## Do I need to open-source my application?

**No.** Using ArcFlow does not impose any license requirement on your application, graph definitions, schemas, workflows, queries, data, models, or generated outputs. You can ship your app under any license (proprietary, MIT, Apache, GPL — whatever you choose), and ArcFlow's proprietary nature does not propagate.

This is the LGPL/MIT-cookbook pattern. The same way using SQLite, Postgres client libraries, or the Linux kernel API doesn't force you to open-source your application — using ArcFlow's runtime doesn't force you to either.

## Are the examples and cookbook MIT-licensed?

**Yes.** Everything in this repository — `cookbooks/`, `examples/` (under `cookbooks/`), `typescript/` (SDK source), `mcp/` (MCP server reference), `react/` (React bindings), `install/`, `docs/`, `schemas/` — is MIT-licensed unless individually marked otherwise.

You can copy any cookbook recipe wholesale into your project, modify it freely, redistribute it, and ship it without attribution beyond the MIT requirement (include the license text or a notice).

## Is the ArcFlow Core engine open source?

**No.** ArcFlow Core (the engine implementation — graph kernel, GraphBLAS algorithms, vector index, GPU kernels, sharded SWMR substrate, daemon, language bindings' native libraries) is **proprietary, free-to-use** software.

We are precise about this because phrasing matters: VS Code's documentation explicitly distinguishes "built on open source" from "open source itself" — we do the same. ArcFlow is not open source; it is free-to-use proprietary software with an open ecosystem around it.

**What you can do without source access:**

- Read the public [JSON-RPC protocol spec](./docs/protocol/jsonrpc-v1.md) and reimplement the API surface
- Verify every release binary via [sigstore attestations](./docs/reproducible-build.md) (`gh attestation verify <asset> --owner ozinc`)
- See the SHA256SUMS file shipped with every release
- Inspect the architectural surface via the cookbook + SDK source

## Who owns my data, graphs, queries, and application code?

**You do.** You retain all rights to:

- Application code you write that uses ArcFlow
- Graph schemas, node types, and edge definitions you author
- Cypher queries you write
- Data you load into ArcFlow
- Output, results, exports, snapshots generated by ArcFlow on your behalf
- Standing-query subscriptions, materialized views, and indexed derivatives

ArcFlow claims no ownership over any of these. We mint no telemetry. We make no callhome connections. Verify with: run the daemon in an air-gapped environment — it works.

## When do I need to pay?

A paid ArcFlow plan is required for one or more of the following, **and only these**:

- **Managed ArcFlow Cloud** — hosted daemon instances, automatic backups, observability dashboards
- **Distributed production clusters above the Free Use Limits** — multi-node SWMR replication, federated workspaces, geo-replicated reads
- **Premium algorithm packs** — advanced spatial/temporal/graph algorithms beyond the open-cookbook baseline
- **Enterprise governance** — SSO (SAML, OIDC), RBAC with audit logs, compliance reports (SOC 2, ISO 27001 boundary)
- **Private support, SLA** — direct engineering support, response-time guarantees, custom deployment assistance
- **Custom licensing** — redistribution outside the standard terms (e.g., re-selling ArcFlow as part of a competing product would require a separate license)

For exact tier definitions, see [oz.com/pricing](https://staging.oz.com/pricing).

## Can I redistribute the ArcFlow binary with my application?

**Yes**, under the standard redistribution terms:

- You may bundle the ArcFlow runtime binaries (`arcflow`, `arcflow-daemon`, `arcflow-mcp`, language-binding `.dylib` / `.so` / `.dll`) inside your application package
- You may not modify the binaries, remove copyright notices, bypass license-control checks (there aren't any in the free tier — but the clause covers future paid-tier additions), or offer ArcFlow itself as a competing standalone product
- Attribution: include the LICENSE file or a notice referring to it in your application's distribution

This means you can ship an Electron app, a Tauri desktop app, a Docker image, a homebrew formula, a Snap package, or anything else that contains ArcFlow's runtime. Reference: how Adobe ships V8 inside Photoshop's scripting layer, how 1Password ships Sentry inside the desktop app.

## What happens if ArcFlow's licensing terms change?

**Released versions stay under the license they shipped under.** If you deployed ArcFlow v1.6.86 in production today, you can continue using v1.6.86 under today's terms forever.

Material pricing or licensing changes apply only to **future versions** (newer than the change-announcement date) and will be announced **at least 180 days in advance** on this repo's release notes + via the [oz.com/pricing](https://staging.oz.com/pricing) page.

This is the explicit anti-Unity-runtime-fee clause. Trust is hard to rebuild; predictability is cheap to commit to.

## Why is the engine closed-source if you want developer adoption?

Because:

1. **Performance and algorithmic depth are the product.** ArcFlow ships GPU kernels, reactive incremental view maintenance, bitemporal indexing, SWMR sharding, and other engine internals that took years to build. Open-sourcing them would invite immediate commercial cloning by hyperscalers, eliminating the revenue model that funds further development. Reference: the Redis/Elastic/MongoDB rounds of cloud-hyperscaler value capture.
2. **You don't need source access to build with it.** The JSON-RPC spec, language bindings (SDK source IS open), cookbook examples, and CLI cover every developer use case. CUDA's success proves this.
3. **You don't need source access to verify it.** Every release binary is signed with sigstore provenance attestations. Reproducible-build instructions are public. The protocol surface is documented and stable.
4. **You're never trapped.** The wire protocol is public and Apache-licensed (or CC-BY) — any third party could write a competing daemon implementing the same protocol. The engine is replaceable in principle; that's the lock-in escape hatch.

## How can I verify a downloaded ArcFlow binary is authentic?

Three layers:

1. **SHA256SUMS** — every release ships a `SHA256SUMS` aggregate file. Verify with:
   ```sh
   curl -fsSL https://github.com/ozinc/arcflow/releases/download/v1.6.86/SHA256SUMS \
       | sha256sum -c --ignore-missing
   ```
2. **Sigstore attestation** — every binary has a sigstore provenance attestation. Verify with:
   ```sh
   gh attestation verify arcflow-1.6.86-linux-x86_64-musl.tar.gz --owner ozinc
   ```
   This proves the binary was built by GitHub Actions in `ozinc/arcflow-core` at a specific source SHA. You don't need to trust OZ infrastructure — the proof is in the sigstore transparency log.
3. **Reproducible builds** — for the most paranoid case, rebuild from source yourself per [`docs/reproducible-build.md`](./docs/reproducible-build.md) and verify the SHA256 matches what we shipped.

## Where does ArcFlow tie in with my data sovereignty / GDPR / SOC 2 posture?

- **ArcFlow Core runs in your process / on your infrastructure** — no callhome, no required cloud connection, no required telemetry. All data stays where you put it.
- **ArcFlow Cloud** (paid managed service) — runs in OZ-managed infrastructure under the terms of our [DPA](https://staging.oz.com/legal). If you need data residency in specific regions, that's a paid-tier feature.
- **Telemetry** — there is none in the free tier. The paid managed service has opt-in usage analytics; opt-out is supported. Verify the free tier's network behavior in any sandbox.

The cookbook, SDK source, MCP server, install scripts, and docs are MIT-licensed and welcome contributions. See [`CONTRIBUTING.md`](./CONTRIBUTING.md) for the contribution workflow.
