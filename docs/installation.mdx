# Installation

## Quick install (CLI today)

The canonical install command on macOS + Linux:

```bash
curl -fsSL https://staging.oz.com/install/arcflow | sh
```

This is the only install URL the docs publish. If a stale CLI build surfaces a different URL (any other domain, including a deployment-host alias), prefer the canonical command above; the published install URL is the source of truth.

**What this gives you today:** the `arcflow` CLI binary at `~/.arcflow/bin/arcflow` (or platform equivalent), and that's it — a single executable. The install message will say *"Binaries: arcflow"* (singular) — that is the current shipping scope.

**What this does NOT give you today:** the Python wheel, the npm package, or the Rust crate. Those are **planned for RAM-C2 / 2026-Q3** on their respective public registries (the current shipping status for each binding lives in the install matrix below). Until they publish, the public-registry installs for `oz-arcflow` / `arcflow` will fail with `No matching distribution`. See [Using ArcFlow from Python or Node today](#using-arcflow-from-python-or-node-today) below for the recovery path.

## Upgrading from a pre-2026-05-14 install (the `v1.x` line)

If you have an `arcflow` binary installed from before 2026-05-14 — anything reporting `v1.x.y` when you run `arcflow --version` — your `arcflow upgrade` command **cannot migrate you to the current line**. The v1.x line was retired 2026-05-14 when ArcFlow re-versioned to alpha (current line: `v0.7.x` and `v0.8.x`); legacy `v1.x` binaries hold their own retired upgrade-channel URLs in their compiled bytes and cannot self-correct.

**Recovery: reinstall via the canonical install script.**

```bash
curl -fsSL https://staging.oz.com/install/arcflow | sh
```

(See [Quick install](#quick-install-cli-today) at the top of this page for the URL-stability context — the install URL standardized on `staging.oz.com` is what works today; the production-canonical `oz.com/install/arcflow` resolves once the oz-platform `dev → production` deploy lands, but the docs publish the working URL.)

After reinstall, verify:

```bash
arcflow --version    # should show v0.7.x or v0.8.x (current line)
arcflow upgrade      # works correctly from v0.7.x+; pulls the latest cut
```

**If you're already on `v0.7.x` or `v0.8.x`:** nothing to do. `arcflow upgrade` works correctly on the current line; you can keep using it.

### Why this happens (one paragraph)

ArcFlow moved from `1.x` to `0.x` versioning per an explicit operator directive — the alpha line restarts at `0.x`, with `v1.0` reserved for the first production-ready release. The `arcflow upgrade` code in any binary shipped before that pivot was compiled against the retired `v1.x` release channel; it has no awareness of the line move. The fix is to download a current binary via the install script (above), which then has the corrected upgrade-channel awareness for every future `arcflow upgrade` call. This is a one-time recovery per machine; no further intervention needed.

## Using ArcFlow from Python or Node today

If your project's `pyproject.toml` or `package.json` depends on `oz-arcflow` / `arcflow` and `pip install` / `npm install` returns *"No matching distribution found"*, the cause is the publish gap above — the SDK wheels are not yet on public registries. Until they ship (RAM-C2 / 2026-Q3), two recovery paths work today:

### Path 1 — Editable install against the engine repo (recommended for dev)

The Python wheel can be built and installed editable from the engine source:

```bash
git clone https://github.com/ozinc/arcflow
cd arcflow
cargo build --release -p arcflow-ffi
pip install -e python/   # editable install, picks up the local libarcflow
```

Then in your project, replace the version pin with the local path:

```toml
# Your project's pyproject.toml
[project]
dependencies = [
    # "oz-arcflow>=0.8"   ← swap this line out until public publish
]
```

…and install your project against the local engine repo:

```bash
pip install -e ~/code/arcflow --force-reinstall --no-deps
pip install -e .[dev] --no-deps   # your project, deps already satisfied
```

The `--no-deps` flag stops pip from re-trying public PyPI for `oz-arcflow`; you've already installed it from source.

### Path 2 — Vendored wheel + `find-links`

For workflows that need a reproducible wheel (CI, multi-developer projects, harnesses that test against a frozen engine commit), build the wheel into a vendored `wheels/` directory and pin via `find-links`:

```bash
~/code/arcflow/python/build_wheel.sh           # produce the wheel
mkdir -p ./wheels
cp ~/code/arcflow/python/dist/oz_arcflow-*.whl ./wheels/
```

```toml
# Your project's pyproject.toml
[project]
dependencies = ["oz-arcflow"]

[tool.uv]
find-links = ["./wheels"]
```

Commit the vendored wheel alongside the tests that depend on it; refresh it whenever you bump the engine version.

### What about the CLI?

Path 1 and Path 2 both apply to the **Python / npm SDK** path. The **CLI binary** (`arcflow`, `arcflow-daemon`, `arcflow-mcp`) ships today via the curl install at the top of this page — no source build required. If your workflow only needs the CLI, the curl install is the complete answer. The SDK paths above only apply when your code calls ArcFlow as an in-process library.

## How this page stays accurate

The matrix above is rendered from
[`release-matrix.json`](https://github.com/ozinc/arcflow/releases/latest/download/release-matrix.json),
published by the engine repo on every release. CI in the engine repo asserts
on every commit that:

- A `status: shipped` artifact's registry returns HTTP 200.
- A `status: planned` artifact's registry returns HTTP 4xx (no false advertising).

If the manifest is unreachable at build time, the docs build fails — there is
no fallback to a hand-rolled list. The matrix above is the live data; this
page never ships an install command that the registry can't actually serve.

## Verify installation

```typescript
import { openInMemory } from 'arcflow'

const db = openInMemory()
db.mutate("CREATE (n:Hello {message: 'It works!'})")
const result = db.query("MATCH (n:Hello) RETURN n.message")
console.log(result.rows[0].get('message'))  // "It works!"
db.close()
```

```python
from arcflow import ArcFlow

with ArcFlow() as db:
    db.execute("CREATE (n:Hello {message: 'It works!'})")
    for row in db.execute("MATCH (n:Hello) RETURN n.message"):
        print(row)  # {'message': 'It works!'}
```

The TypeScript example assumes the `arcflow` npm package is installed; the
Python example assumes the `arcflow` Python wheel is installed. The matrix
above shows you the install command for whichever path is currently shipped.

## Environment variables

| Variable | Default | Description |
|---|---|---|
| `ARCFLOW_METAL_FORCE_UNAVAILABLE` | `false` | Set to `true` to disable Metal GPU acceleration (useful if module load hangs) |
| `ARCFLOW_NATIVE_DIR` | unset | Python only. Override the directory containing `libarcflow.{so,dylib,dll}` (used when running against a locally-built engine instead of the wheel-bundled binary). |

## Requirements

- Node.js 18+ (LTS recommended) for the npm path.
- Python 3.8+ for the pip path.
- No native build tools required — pre-built binaries are bundled in the
  installable artifacts.

## Dev-track install (preview / unreleased builds)

Working against an unreleased ArcFlow change — testing a fix the engine
team just merged, or building against `main` ahead of a release —
means building from source. There is no separate staging registry: the
canonical release path is the GitHub Release on `ozinc/arcflow` (created
by the engine repo's `release-binaries.yml` on tag push). Per-commit
pre-release artifacts are not published.

### Local build

To test an unreleased engine change, build the wheel locally and install
it from a path:

```bash
git clone https://github.com/ozinc/arcflow
cd arcflow
cargo build --release -p arcflow-ffi
pip install -e python/  # editable install, picks up the local libarcflow
```

```python
import os
# Point at the locally built shared library:
os.environ["ARCFLOW_NATIVE_DIR"] = "/abs/path/to/arcflow/target/release"
from arcflow import ArcFlow
```

### When to vendor wheels

Two workflows are valid; the distinction is whether the wheel under
test is published or in-progress.

**Customer projects** consume ArcFlow as a published dependency. Pin
`oz-arcflow==X.Y.Z` in `pyproject.toml` against PyPI. Don't vendor
wheels — they go stale on every release, and the next teammate who
clones the repo gets a binary that doesn't match the docs they're
reading.

**Engine-team stress harnesses** (e.g. `project-merlin`) test against
an in-progress wheel built from `arcflow-core` HEAD. Vendor the wheel
into `wheels/` and pin via `--find-links ./wheels`:

```bash
arcflow-core/python/build_wheel.sh    # produce the wheel
cp arcflow-core/python/dist/oz_arcflow-*.whl <harness>/wheels/
```

```toml
# Harness pyproject.toml
[project]
dependencies = ["oz-arcflow"]

[tool.uv]
find-links = ["./wheels"]
```

The vendored wheel rebuilds whenever the engine code changes; commit
it alongside the test that depends on it for reproducible audit across
sessions.
