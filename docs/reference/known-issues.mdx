# Known Issues

## Metal GPU initialization hang

**Symptom:** `require('arcflow')` hangs indefinitely on macOS.

**Cause:** Metal GPU kernel compilation can deadlock during module initialization on some systems.

**Fix:**

```bash
export ARCFLOW_METAL_FORCE_UNAVAILABLE=true
```

This disables Metal acceleration and falls back to CPU. All queries still work — just without GPU acceleration.

## pnpm file: protocol binary sync

**Symptom:** After rebuilding the native module, pnpm doesn't pick up the new `.node` binary.

**Fix:** Manually copy the binary:

```bash
# Manually copy the built .node binary into your node_modules
cp <built-binary>.node node_modules/arcflow/arcflow.<platform>.node
```

## AS OF seq with bound variables

`AS OF seq` accepts a literal or a parameter today. The bound-variable form — using a value carried in from a prior `MATCH` — is not yet wired:

```cypher
-- Works today: literal
MATCH (n:Person) AS OF seq 100 RETURN n.name

-- Works today: parameter
MATCH (n:Person) AS OF seq $approval_seq RETURN n.name

-- Not yet wired: bound variable / arithmetic on a bound variable
MATCH (g:Goal {match_id: 'm1'})
MATCH (b:Ball) AS OF seq g.seq - 10
RETURN b.position
```

**Workaround until the bound-variable form lands:** issue two queries and pass the seq through as a parameter.

```typescript
const goal = db.query("MATCH (g:Goal {match_id: 'm1'}) RETURN g.seq AS seq").rows[0]
const seq = (goal.get('seq') as number) - 10
db.query("MATCH (b:Ball) AS OF seq $seq RETURN b.position", { seq })
```

## Spatial index granularity for polygons and lists

**Symptom:** `MATCH (z:Zone) WHERE withinDistance(z.shape, $point, 100)` on `Polygon` or `LineString` properties is correct but slower than the equivalent query on `Point` properties at the same node count.

**Cause:** The ArcFlow Spatial Index currently entries `LineString` and `Polygon` values by their **centroid** only. Radius / bbox / KNN queries return a centroid-based candidate set; exact containment and intersection are then computed at query time against the full geometry of each candidate. `Vector3d` and `Extent` are stored but not spatially indexed at all — predicates over them fall back to a full property scan.

`Point` and `Point3d` are indexed exactly, so radius and KNN queries on point-valued properties run at the full ArcFlow Spatial Index speed.

**Workaround until full-geometry indexing lands:** for hot-path queries over polygon / linestring properties, store an additional `Point` property (e.g. `centroid` or `representative_point`) and predicate against that. The spatial index will use it directly; you can still apply exact-geometry filters as a follow-on `WHERE` clause.

Tracked engine-side as `wave-SPAT-0001`.
