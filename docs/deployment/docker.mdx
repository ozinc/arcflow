# Why ArcFlow doesn't ship a Docker image

ArcFlow is a 5MB statically-linked library. Wrapping it in a Docker image
adds ~20MB of container overhead and subverts the central design point —
the engine runs **in your process**, not in a sidecar.

This page exists because it's the question we get asked. The answer is in
the engine release manifest:

```toml
# RELEASE-MATRIX.toml — engine repo
[[artifacts]]
kind = "docker"
status = "refused"
```

`status: refused` is a first-class lifecycle state. It signals a deliberate
decision not to ship, with a recorded reason. The same release manifest
drives the live install matrix on the [Installation page](/installation),
so this is never a stale claim — it's what the engine actually ships.

## What to use instead

| Use case | Right shape |
|---|---|
| Embedded in a Python / Node / Rust application | The matching language binding from the [Installation page](/installation) — the engine runs in your process |
| One-shot scripting or CLI workflow | The native CLI binary from the [Installation page](/installation) — 5MB, runs anywhere |
| Zero-install kick-the-tires demo | [WASM playground](/engine) — runs the same engine in your browser |
| Long-running `arcflow-mcp` service for cloud LLM clients | See [MCP Server](/mcp) — runs as a child process of your LLM client, stdin/stdout JSON-RPC |

## When this might change

If a future use case needs a containerized shape — a shared multi-tenant
deployment, an `arcflow-mcp` service that benefits from isolation, an
edge-cache image — it will land as a **separate** artifact kind in the
manifest (e.g., `kind = "mcp-docker"` or `kind = "service-docker"`), not as
a revival of generic engine Docker.

The discipline matters. Conflating "the engine binary in a container" with
"a service that happens to use the engine" is what made the original
install matrix lie to customers in the first place. The manifest tracks
shape, not aspiration.
