# Temporal Queries

**Time is a first-class dimension of the world model — not a log, not an audit table, not a framework to integrate. Every state the graph has ever been in is directly queryable.**

For autonomous systems, this is the difference between learning and guessing. A robot that made a wrong navigation decision can reconstruct its exact world model state at the moment of the decision — not a log entry, not a diff, the full spatial-temporal graph as it existed then. That is what makes safety analysis, model debugging, and decision auditing tractable rather than speculative.

Every mutation in ArcFlow is versioned and preserved. The world model at any previous moment is accessible with a single `AS OF seq N` clause on any `MATCH` — no schema changes, no separate tables, no event-sourcing pipeline. Point-in-time queries, deterministic replay, temporal comparison, and clock domain merging are built into the query language itself.

```cypher
-- Current state
MATCH (o:Order {id: 'ORD-001'}) RETURN o.status  -- "shipped"

-- State at mutation sequence 100 (before the shipment mutation)
MATCH (o:Order {id: 'ORD-001'}) AS OF seq 100 RETURN o.status  -- "confirmed"
```

## Real-world queries

The power of temporal queries is not in the syntax — it's in the questions you can now ask that were previously unanswerable without a separate audit system.

**Post-incident reconstruction: who was where when it happened?**

```cypher
-- All players within 3m of the ball, 10 sequences before the goal was scored
MATCH (g:Goal {match_id: 'match-001'})
MATCH (b:Ball) AS OF seq g.seq - 10
  WHERE distance(b.position, g.position) < 1.0
MATCH (p:Player) AS OF seq g.seq - 10
  WHERE distance(p.position, b.position) < 3.0
RETURN p.name, p.team, p._confidence
ORDER BY p._confidence DESC
```

The world model at `g.seq - 10` is the exact graph state ten mutations before the goal was recorded. No log reconstruction. No event replay pipeline. The same query syntax, a different time.

**Fraud investigation: what did we know before we approved it?**

```cypher
-- Account state at the moment the transfer was authorised
MATCH (a:Account {id: $account_id}) AS OF seq $approval_seq
RETURN a.balance, a.risk_score, a.last_verified_at, a._confidence

-- Which facts were predicted (not observed) at that moment?
MATCH (a:Account {id: $account_id})-[r:LINKED_TO]->(b:Account)
  AS OF seq $approval_seq
WHERE r._observation_class = 'predicted'
RETURN b.id, r._confidence, r.source
```

**Autonomous system debugging: what did the world model show before the decision?**

```cypher
-- Robot's view of obstacle positions 500ms before the collision
MATCH (o:Obstacle) AS OF seq $collision_seq - 50
RETURN o.name, o.position, o._observation_class, o._confidence
ORDER BY o._confidence ASC

-- Confidence at the moment versus now — did confidence degrade?
MATCH (o:Obstacle {id: $id}) AS OF seq $collision_seq - 50
RETURN o._confidence AS confidence_then

MATCH (o:Obstacle {id: $id})
RETURN o._confidence AS confidence_now
```

**World model staleness: which facts aged past their reliable window?**

```cypher
-- Facts that were current at a past checkpoint but have since been superseded
MATCH (f:Fact) AS OF seq 5000
WHERE f._observation_class = 'observed'
WITH f.id AS fact_id, f._confidence AS old_confidence

MATCH (f:Fact {id: fact_id})
WHERE f._confidence < old_confidence - 0.2
RETURN fact_id, old_confidence, f._confidence AS current_confidence
ORDER BY (old_confidence - f._confidence) DESC
```

These are not reconstructions from logs. They are direct queries against the versioned world model — same graph, same language, different point in time.

## Clock and Ticks

The engine maintains a logical clock. Each mutation advances the clock tick.

```cypher
CALL db.clock()
```

```
| field | value      |
|---
----|---
------
---|
| tick  | 1700000042 |
```

### ArcFlow Clock Domains

Real-world data arrives from sources that run at different rates — a camera at 25Hz, a LIDAR at 10Hz, an RFID reader at 1Hz. **ArcFlow Clock Domains** let each source maintain its own independent temporal sequence while the world model merges them into a coherent view. No external time-series middleware, no alignment preprocessing — each domain advances at its own rate and the graph stays consistent.

```cypher
CALL db.clockDomains()
```

Each domain reports its current tick, last mutation timestamp, and source identity. Observations ingested from different clock domains carry their originating domain in provenance metadata — so a query can distinguish a LIDAR reading from tick 1420 and a camera reading from tick 356, even when both describe the same entity at the same wall-clock instant.

### Querying AS OF a domain tick

A clock domain can be made **queryable**: register it, seal its frontier as data arrives, and then time-travel by a domain-meaningful tick — a camera frame, a software-clock tick — instead of a raw WAL sequence. The engine owns the tick→sequence mapping (one writer maintains it at ingest; no client ever keeps a mirror), so the same machinery that powers `AS OF seq N` answers `AS OF <domain> <tick>`.

```cypher
-- Register a queryable domain, then seal frontier points as data lands.
CALL clock.register('swclock.tick')
CALL clock.advance('swclock.tick', 20) YIELD domain, tick, sealed
```

```cypher
-- Time-travel by the domain's own tick. Resolution is floor-with-disclosure:
-- the engine pins to the greatest sealed tick <= the one you asked for.
MATCH (n:Person) AS OF swclock.tick 15 RETURN n.name
```

The tick form accepts a bare name (`tick`), a dotted name (`swclock.tick`), or a `$param`. It inherits the narrow `AS OF` grammar — a single label, `RETURN`-only — and `AS OF seq N` is unchanged.

Resolve a tick to its pinned sequence for diagnostics — also available on the daemon wire as the additive `clock.resolve` JSON-RPC method:

```cypher
CALL clock.resolve('swclock.tick', 15) YIELD domain, resolved_tick, seq
```

#### Typed errors

| Code | When |
|---|---|
| `TICK_AHEAD_OF_FRONTIER` | The tick is newer than the latest sealed frontier — that answer isn't sealed yet. |
| `TICK_BELOW_FIRST_SEAL` | The tick predates the first seal point for the domain. |
| `CLOCK_DOMAIN_REGRESSION` | An advance tried to move the frontier backwards. |
| `CLOCK_DOMAIN_UNKNOWN` | The domain was never registered. |
| `CLOCK_DOMAIN_INVALID_NAME` | The name isn't `[A-Za-z_][A-Za-z0-9_.]*`, or collides with `seq` / a query keyword. |

> **Availability:** the queryable clock-domain surface ships in engine 0.10.27 (EDGE-A4). `clock.register` / `clock.advance` ride `cypher.execute`; `clock.resolve` is an additive daemon method. Durable WAL capture of the registry across restart lands in a follow-up wave (EDGE-A2) — until then treat a domain's seals as session-scoped.

## AS OF Queries

Query the graph as it existed at a specific **mutation sequence number**. The engine replays the WAL onto a clean snapshot at that sequence and executes the query against it.

```cypher
-- Preferred: AS OF seq N (exact, WAL-based snapshot)
MATCH (n:Person) AS OF seq 42 RETURN n.name
```

```cypher
-- Create nodes — each mutation advances the sequence
CREATE (a:Person {name: 'Alice'})  -- seq 1
CREATE (b:Person {name: 'Bob'})    -- seq 2

-- Query the graph at seq 1 — only Alice existed
MATCH (n:Person) AS OF seq 1 RETURN n.name
```

```
| name  |
|---
----|
| Alice |
```

Only mutations up to and including sequence N are visible.

> **Sequence is the temporal axis.** `AS OF` takes a WAL sequence (`AS OF seq N`) — exact and replayable. Wall-clock timestamps are not a temporal axis: `AS OF <timestamp>` is rejected (`DEPRECATED_AS_OF_TIMESTAMP`).

### Nodes As Of

Procedure form for temporal point-in-time queries:


## Temporal Ingest

Insert nodes with explicit timestamps, independent of the current clock:

```typescript
// Ingest a historical fact with an explicit timestamp
db.mutate(
  "CREATE (e:Event {name: $name}) AS OF $ts",
  { name: 'Launch', ts: 1609459200 }  // 2021-01-01 00:00:00 UTC
)
```

Temporal ingest is used for backfilling historical data, replaying event streams, and merging data from external systems with their original timestamps.

## Temporal Compare

Compare the graph state between two points in time:

```cypher
CALL db.temporalCompare()
```

Returns the diff: nodes added, removed, and properties changed between two temporal snapshots.

## Temporal Replay

Replay mutations in chronological order from a starting tick:

```cypher
CALL db.temporalReplay()
```

Useful for auditing, debugging, and reconstructing how the graph evolved over time.

### Deterministic replay (the contract)

`AS OF seq N` is not just a feature — it is a **contract**. The same `(query, seq)` executed against the same pinned `(engine_version, plan_version)` yields a **byte-identical canonicalized result**, across a process restart and across a rebuild. Replay is reproducible, so a past answer can be re-derived and audited rather than trusted.

**What "canonicalized" means.** The guarantee is over a canonical digest of the result, not incidental wire bytes:

- rows are **totally ordered by `_id`** before hashing (not scan order);
- cell strings are hashed **verbatim** (no re-parse, no reformat);
- wall-clock-stamped keys (`_created_at`, `_updated_at`) are **excluded** — they are not deterministic on replay;
- everything else, including `_confidence`, `_out_degree`, `_in_degree`, and `_rel_types`, **is** deterministic on replay and is part of the digest.

**Per-version, by design.** Identity holds *within* a pinned `(engine_version, plan_version)`. Across versions the digest **may** differ — and that is intentional, not a bug. Both versions ride a separate field on the result envelope, so any change in a replayed answer is **attributable** to a version bump rather than a silent identity break. There is no cross-version identity promise.

> **Availability:** the contract holds at the engine level today, enforced by an in-tree conformance test (FF-EDGE-06). The `(plan_version, engine_version)` accessor on the result envelope is **not yet exposed** on the daemon wire or the language SDKs — that exposure lands in a follow-up wave (EDGE-A2). Rely on the replay *semantics* now; reach for the on-result accessor when A2 ships it.

## Monotonic Validation

ArcFlow enforces monotonic clock progression within a single engine instance. Timestamps never go backward. This guarantees causal ordering: if fact A was created before fact B, A's timestamp is strictly less than B's.

External timestamps from `temporal_ingest` are stored as-is but are clearly marked as externally sourced. The engine's own clock remains monotonic.

## Stream Merging

When merging data from multiple sources with independent clocks:

1. Each source's timestamps are preserved in their original clock domain
2. Conflicts are detected when two sources assert contradictory facts at overlapping time ranges
3. The merge result includes provenance metadata identifying which source contributed each fact
