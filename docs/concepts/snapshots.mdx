# Snapshot-Pinned Reads

Every world model query observes a specific snapshot of the graph. ArcFlow makes that
snapshot **citeable, round-trippable, and re-queryable** — every result envelope carries
a `__snapshot` URI, and any past snapshot can be replayed against an identical query and
return identical bytes.

This is what makes a world model trustworthy under scrutiny: the customer who got an
answer can ask *"which world state did this answer come from?"* and the engine has a
one-line answer.

## The snapshot URI

Every committed snapshot has a content-addressed URI of the form:

```
arcflow://snapshot/<hex-hash>
```

The hash is computed deterministically over the write set. Identical writes produce
identical URIs across processes and machines — two replicas that ingested the same WAL
agree on snapshot URIs without coordination.

## Every result carries a snapshot

`db.query()`, the CLI, the HTTP server, and the MCP server all return the snapshot URI
that the query observed. Provenance is automatic — there's nothing extra to opt into.

```typescript
const result = db.query("MATCH (e:Entity) RETURN e.name, e._confidence")
result.snapshotUri    // "arcflow://snapshot/9c3b…"  — this is what the query saw
result.rows           // the actual result rows
```

The CLI emits the same field in its JSON envelope:

```bash
arcflow query "MATCH (n) RETURN count(*)" --json
```
```json
{
  "__snapshot": "arcflow://snapshot/9c3b8a1f7d2e…",
  "columns": ["count(*)"],
  "rows": [{"count(*)": "10234"}]
}
```

The HTTP server returns it as headers:

```
X-Arcflow-Snapshot-Id: arcflow://snapshot/9c3b8a1f7d2e…
X-Arcflow-Manifest-Pin: arcflow://snapshot/9c3b8a1f7d2e…
```

The MCP `read_query` and `graph_rag` tools include `snapshot_id` in their JSON response
envelope.

## `query_at` — replay against a pinned snapshot

Once you have a snapshot URI, re-run the same query against that exact world state:

```typescript
// Original query
const a = db.query("MATCH (e:Entity {id: 'unit-01'}) RETURN e._confidence")
const uri = a.snapshotUri

// ... time passes, more mutations happen ...

// Re-run the same query against the original world state
const b = db.queryAt(uri, "MATCH (e:Entity {id: 'unit-01'}) RETURN e._confidence")
// b.rows is bit-for-bit identical to a.rows
```

The Rust SDK returns `TaggedQueryResult { snapshot_uri, rows }`. The TypeScript and
Python wrappers expose the same shape on the result object.

## CLI: `--at-snapshot` and `arcflow rerun`

Pin any CLI query to a historical snapshot:

```bash
arcflow query "MATCH (e:Entity {id: $id}) RETURN e._confidence" \
  --param id=unit-01 \
  --at-snapshot arcflow://snapshot/9c3b8a1f7d2e…
```

`arcflow rerun` is a thin convenience wrapper that always pins:

```bash
arcflow rerun \
  --snapshot arcflow://snapshot/9c3b8a1f7d2e… \
  --query "MATCH (e:Entity {id: 'unit-01'}) RETURN e._confidence"
```

Both surface the same JSON envelope with the same `__snapshot` field — and the field
matches the URI you pinned, confirming the engine actually executed against that
snapshot rather than silently falling forward.

## HTTP: `?at=` query parameter

Pin reads over the HTTP server with the `at` query string:

```
GET /query?cypher=MATCH%20(n)%20RETURN%20count(*)&at=arcflow://snapshot/9c3b…
```

The response carries the same `X-Arcflow-Snapshot-Id` and `X-Arcflow-Manifest-Pin`
headers as un-pinned reads — easy to verify in any HTTP client.

## MCP: `snapshot_uri` input

The MCP `read_query` and `graph_rag` tools accept an optional `snapshot_uri` input. If
provided, the tool runs against that snapshot and returns it back in the response
envelope. If omitted, the engine reads the latest snapshot and returns its URI as
before.

## Why this matters — the provenanced answer

The customer story is simple. An agent or a human got a surprising answer from a world
model query. Without snapshot pinning, the only investigation path is *"run the query
again and hope the data hasn't changed."* With snapshot pinning, the answer is
deterministic: take the URI off the original result, paste it into `arcflow rerun`, get
the same answer back. If the answer differs, that itself is a finding — the data has
moved, and you have both URIs to compare.

This is what makes ArcFlow safe to use behind a customer-facing answer pipeline. Every
output is provenanced. Every claim is reproducible. There is no "works on my machine"
for world model state.

## When to use

- You need reproducible query results in a pipeline where the world model is being written concurrently — pin the snapshot URI after the first query and use `db.queryAt(uri, ...)` for all subsequent queries in that pipeline run.
- An AI agent or human analyst reports a surprising answer from a world-model query; take the `__snapshot` URI from the result envelope and `arcflow rerun` it to confirm whether the answer was correct at that moment.
- You are building a compliance or audit system that must demonstrate what the world model showed at a specific decision point — the snapshot URI is the citation.
- You need to compare two versions of the world model deterministically (e.g., before and after a data migration) — compute fingerprints of both snapshots and diff the URIs.
- Cross-service pipelines where service A queries ArcFlow and passes results to service B — include the snapshot URI in the payload so service B can re-query the exact same world state if it needs to validate A's results. <!-- DRAFT: confirm whether snapshot URIs are stable across WAL compaction and checkpoint -->

## When not to use

- Snapshot URIs are not persistent bookmarks for human-facing URLs. They are content-addressed references valid for the lifetime of the WAL segments that produced them. After WAL compaction or a database restore, a URI from an old WAL generation may not be resolvable. <!-- DRAFT: confirm WAL retention policy and snapshot URI validity window -->
- Do not use `db.queryAt(uri, ...)` as a general-purpose "read-only mode". It is intended for point-in-time re-query of a specific past state, not for broad historical browsing. For that, use `AS OF seq N` temporal queries directly.
- The `X-Arcflow-Snapshot-Id` HTTP header is informational — it tells you what snapshot the server observed, but it does not guarantee that the server's world model has not advanced since the request. Do not use the header as a freshness check.
- Do not rely on snapshot URIs for deduplication of write operations. Two identical writes to different databases produce the same URI, but the same write to the same database at different times produces different URIs. The hash is over the write set, not the resulting graph state.
- If your query does not care about historical reproducibility (e.g., a live dashboard showing current entity positions), snapshot pinning adds latency and complexity for no benefit. Use live queries or plain `db.query()`.

## Failure modes

- **URI extracted from the wrong result envelope**: a pipeline that runs five sequential queries but only records the snapshot URI from the first one will pin re-runs to that early state. If the intent was to capture the state after all five mutations, the URI is wrong and `arcflow rerun` will return stale data with no error.
- **Snapshot URI silently ignored by older client**: if a TypeScript or Python wrapper predating snapshot URI support is used, `result.snapshotUri` is `undefined` and accessing it does not throw. Code that logs or forwards `undefined` as the URI may produce invalid pins downstream.
- **`arcflow rerun` on a compacted snapshot**: if the WAL segment containing the snapshot has been compacted and the snapshot URI is no longer resolvable, `arcflow rerun` fails with an error. The error message should reference the URI; if it does not, the failure is hard to diagnose. <!-- DRAFT: confirm exact error code / message for unresolvable snapshot URIs -->
- **Concurrent writes shifting the snapshot between check and re-query**: a caller reads `result.snapshotUri` and then issues a mutation before calling `db.queryAt(uri, ...)`. The mutation does not affect the pinned query — but if the caller expected to see the mutation in the re-query, it will not. This is a misunderstanding of snapshot semantics, not a bug, but it produces silent result discrepancies.
- **MCP `snapshot_uri` input field ignored when not provided**: the MCP `read_query` tool accepts an optional `snapshot_uri`. If the caller omits it, the tool reads the latest snapshot — which is almost always correct, but means two calls to the same MCP tool within the same agent turn may observe different snapshots if a concurrent writer is active.
