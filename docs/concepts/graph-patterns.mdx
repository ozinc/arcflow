# Graph Patterns

In SQL you describe *conditions on rows*. In GQL you describe *shapes in the graph* — and the engine finds everything that matches that shape.

This is the shift. It changes not just the syntax but how you think about the problem.

## What a pattern is

A pattern is a template for a subgraph. It specifies nodes, the relationships between them, and any constraints on their properties. The engine scans the graph and returns every subgraph that fits the template.

```cypher
-- "Find every Person connected to a Company by a WORKS_AT relationship"
MATCH (p:Person)-[:WORKS_AT]->(c:Company)
RETURN p.name, c.name
```

The `MATCH` clause holds the pattern. Everything inside it — `(p:Person)`, `-[:WORKS_AT]->`, `(c:Company)` — describes a shape, not a filter condition.

## Node patterns

A node pattern describes a single node:

```
(variable:Label {property: value})
```

| Element | Meaning |
|---|---|
| `(n)` | Any node, bound to variable `n` |
| `(n:Person)` | Any node labeled `:Person` |
| `(n:Person {name: 'Alice'})` | A Person node where `name` equals `'Alice'` |
| `(:Person)` | A Person node (anonymous — no variable needed) |

```cypher
-- Any node
MATCH (n) RETURN n

-- Labeled node
MATCH (n:Person) RETURN n.name

-- Labeled with property filter
MATCH (n:Person {city: 'Reykjavik'}) RETURN n.name

-- Anonymous (you don't need the value, just to confirm the pattern exists)
MATCH (:Company {name: 'Acme'})-[:HAS]->(d:Department) RETURN d.name
```

## Relationship patterns

A relationship pattern connects two nodes:

```
(a)-[r:TYPE {property: value}]->(b)
```

| Element | Meaning |
|---|---|
| `-[:KNOWS]->` | Directed relationship of type KNOWS |
| `<-[:KNOWS]-` | Reverse direction |
| `-[:KNOWS]-` | Either direction |
| `-[r:KNOWS]->` | Bind to variable `r` to access properties |
| `-[r]->` | Any relationship type |
| `-[:KNOWS {since: 2020}]->` | Relationship with property filter |

```cypher
-- Directed
MATCH (a:Person)-[:KNOWS]->(b:Person) RETURN a.name, b.name

-- Either direction
MATCH (a:Person)-[:KNOWS]-(b:Person) RETURN a.name, b.name

-- With relationship properties
MATCH (a)-[r:KNOWS]->(b) WHERE r.since > 2019 RETURN a.name, b.name, r.since

-- Any relationship type
MATCH (a:Person)-[r]->(b) RETURN a.name, type(r), b.name
```

## Variable-length paths

Append `*min..max` to a relationship pattern to match paths of varying depth:

```cypher
-- Exactly 2 hops
MATCH (a)-[:KNOWS*2]->(b) RETURN b.name

-- 1 to 3 hops
MATCH (a:Person {name: 'Alice'})-[:KNOWS*1..3]->(b) RETURN b.name

-- Any number of hops (use carefully on large graphs)
MATCH (a)-[:KNOWS*]->(b) RETURN b.name
```

This is where the power of graph patterns becomes obvious. The SQL equivalent requires a recursive CTE or a series of self-joins — one per hop depth. In GQL, you write it once.

```sql
-- SQL: "who does Alice know, up to 3 hops?"
WITH RECURSIVE knows AS (
  SELECT person_id, known_id, 1 as depth FROM relationships WHERE person_id = alice_id
  UNION ALL
  SELECT k.person_id, r.known_id, k.depth + 1
  FROM knows k JOIN relationships r ON k.known_id = r.person_id
  WHERE k.depth < 3
)
SELECT DISTINCT known_id FROM knows;

-- GQL: same thing
MATCH (alice:Person {name: 'Alice'})-[:KNOWS*1..3]->(b) RETURN DISTINCT b.name
```

## Multi-pattern (implicit join)

Multiple `MATCH` clauses in sequence find nodes that satisfy all patterns simultaneously:

```cypher
-- Find a person AND a company independently (Cartesian product)
MATCH (p:Person {name: 'Alice'})
MATCH (c:Company {name: 'Acme'})
RETURN p.name, c.name

-- Find people who work at the same company as Alice
MATCH (alice:Person {name: 'Alice'})-[:WORKS_AT]->(c:Company)
MATCH (colleague:Person)-[:WORKS_AT]->(c)
WHERE colleague.name <> 'Alice'
RETURN colleague.name, c.name
```

## OPTIONAL MATCH

Like a SQL LEFT JOIN — returns the pattern match if it exists, `null` if it doesn't:

```cypher
MATCH (p:Person)
OPTIONAL MATCH (p)-[:HAS_PHONE]->(ph:Phone)
RETURN p.name, ph.number  -- ph.number is null if no phone exists
```

## Patterns in context: the mental shift from SQL

In SQL, a many-to-many relationship requires a junction table and two JOINs. In GQL, it's just a traversal:

```sql
-- SQL: "find people who like the same movies"
SELECT a.name, b.name
FROM people a
JOIN likes al ON a.id = al.person_id
JOIN likes bl ON al.movie_id = bl.movie_id
JOIN people b ON bl.person_id = b.id
WHERE a.id <> b.id;
```

```cypher
-- GQL: same question, no junction table needed
MATCH (a:Person)-[:LIKES]->(m:Movie)<-[:LIKES]-(b:Person)
WHERE a <> b
RETURN a.name, b.name, m.title
```

The GQL pattern directly mirrors how you'd draw the relationship on a whiteboard. The SQL query is an artifact of how the data is stored.

## ArcFlow extensions to patterns

WorldCypher adds pattern capabilities beyond the GQL standard:

**Confidence filtering on relationships:**
```cypher
-- Only traverse edges the system is confident about
MATCH (a:Person)-[r:KNOWS]->(b)
WHERE r._confidence > 0.8
RETURN a.name, b.name, r._confidence
```

**Temporal patterns — the graph as it existed before:**
```cypher
-- Who did Alice know 30 days ago?
MATCH (alice:Person {name: 'Alice'})-[:KNOWS]->(b) AS OF seq 1000
RETURN b.name
```

**Live patterns — re-evaluate when the graph changes:**
```cypher
-- Standing query: fires when a new high-score person appears
LIVE MATCH (n:Person) WHERE n.score > 0.95 RETURN n.name, n.score
```

## When to use

- Use `MATCH` patterns whenever you need to express a relationship between entities — they are always more readable and more performant than equivalent JOIN chains for connected data.
- Use variable-length paths (`[:KNOWS*1..3]`) when the number of hops between entities is not fixed or not known in advance — the typical case in social graphs, dependency trees, and communication topology analysis.
- Use `OPTIONAL MATCH` when a relationship may or may not exist and you still want to return the base node — the GQL equivalent of a LEFT JOIN, without requiring a junction table.
- Apply ArcFlow's `_confidence` filter on relationship patterns (`WHERE r._confidence > 0.8`) when traversing a world model where edge quality is heterogeneous — this prunes low-signal paths before returning results.
- Use `LIVE MATCH` (standing patterns) when your application must react to new matches the instant they appear in the graph — live monitoring, alert pipelines, and task-assignment systems all benefit from this pattern.

## When not to use

- Do not use unbounded variable-length paths (`[:KNOWS*]`) on large graphs without a `LIMIT` or a tight `WHERE` predicate. Every reachable node is a candidate match; on a dense graph this produces Cartesian explosion in result size and can exhaust memory. <!-- DRAFT: confirm whether the engine enforces a configurable hop-depth cap on unbounded paths -->
- The multi-pattern syntax (`MATCH (a) MATCH (b)` without a shared variable) produces a Cartesian product. If both patterns match thousands of nodes independently, the result set will have millions of rows. Always share at least one variable between consecutive `MATCH` clauses unless a Cartesian product is intended.
- `LIVE MATCH` subscriptions are not appropriate for patterns that match the majority of the graph. Standing queries are maintained incrementally on every mutation — a pattern that matches 90% of nodes will fire on nearly every write and degrade write throughput.
- GQL patterns are not suited for fuzzy structural matching (e.g., "find nodes that are similar to this subgraph"). Patterns are exact — label, type, and direction are hard constraints. For structural similarity, combine graph patterns with vector search.
- Do not use anonymous nodes (`(:Label)`) in mutation clauses (`CREATE`, `MERGE`). Anonymous nodes cannot be referenced later in the same query and produce nodes that are immediately orphaned from the variable binding scope.

## Failure modes

- **Direction mismatch produces empty results**: `MATCH (a)-[:WORKS_AT]->(c:Company)` and `MATCH (a)-[:WORKS_AT]-(c:Company)` return different results when the stored edge direction is opposite to what the pattern expects. Direction errors are silent — the engine returns zero rows, not an error. Always verify edge direction with `MATCH ()-[r:TYPE]->() RETURN count(r)` before building traversal queries.
- **Implicit Cartesian product from unrelated MATCH clauses**: a query with `MATCH (p:Person) MATCH (c:Company)` where neither `p` nor `c` is constrained to each other returns every person paired with every company. On a graph with 10K persons and 1K companies this produces 10M rows. No warning is emitted.
- **Property filter on missing property returns empty**: `MATCH (n:Person {city: 'Reykjavik'})` returns zero rows if no `:Person` node has a `city` property at all (versus having `city` set to a different value). There is no distinction between "property absent" and "property has a different value" in the match result.
- **Variable reuse across MATCH clauses binds unexpectedly**: if the same variable name is used in two `MATCH` clauses (`MATCH (n:A) MATCH (n:B)`), GQL treats it as a single binding — `n` must satisfy both labels simultaneously. If the intent was two independent nodes, use different variable names.
- **`LIVE MATCH` fires on its own mutations**: an agent that subscribes to a pattern and then writes mutations that change that pattern will receive its own mutations as events. If the handler naively re-writes on every event, it can create a feedback loop. Handlers must be idempotent and guard against self-triggered re-entry. <!-- DRAFT: confirm whether the engine provides a built-in way to suppress self-generated events in live subscriptions -->
