# Observations & Evidence

The **ArcFlow Evidence Model** classifies every fact in the graph by how it was obtained. Each node and relationship carries three pieces of epistemic context: an observation class (how it was acquired), a confidence score (how reliable it is), and a provenance chain (which source, skill, or model produced it). This lets queries filter not just on *what* is known, but on *how well* it is known — the foundation for trusted RAG, confidence-gated coordination, and audit trails that survive scrutiny.

## Three observation classes

| Class | Meaning | Example |
|---|---|---|
| **Observed** | Directly measured from a sensor or data source | "Camera 3 detected person at (52.3, 34.1)" |
| **Inferred** | Derived from observations via rules or algorithms | "Person is likely Alice based on ReID embedding" |
| **Predicted** | Forecast from models or extrapolation | "Person will be at (53.0, 34.5) in 2 seconds" |

## Query by observation class

```cypher
-- All observed nodes (direct evidence)
CALL db.nodesByObservation.observed()

-- All inferred nodes (derived knowledge)
CALL db.nodesByObservation.inferred()

-- All predicted nodes (forecasts)
CALL db.nodesByObservation.predicted()

-- List available observation classes
CALL db.observationClasses()
```

## SDK usage

```typescript
import { open } from '@ozinc/arcflow'

const db = open('./evidence-graph')

// Create an observed entity (from sensor)
db.mutate("CREATE (p:Person {id: 'det-42', _observation_class: 'observed', _confidence: 0.95, source: 'camera-3', x: 52.3, y: 34.1})")

// Create an inferred entity (from algorithm)
db.mutate("CREATE (id:Identity {id: 'reid-alice', _observation_class: 'inferred', _confidence: 0.87, source: 'reid-model', person_id: 'det-42'})")

// Create a predicted entity (from model)
db.mutate("CREATE (pred:Position {id: 'pred-42-t2', _observation_class: 'predicted', _confidence: 0.72, source: 'kalman-filter', x: 53.0, y: 34.5, t_offset: 2})")

// Query only observed data (highest trust)
const observed = db.query("CALL db.nodesByObservation.observed()")
console.log(`Observed entities: ${observed.rowCount}`)

// Query inferred data
const inferred = db.query("CALL db.nodesByObservation.inferred()")
```

## Confidence propagation

Every node carries a confidence score. When you traverse relationships, confidence flows through the graph:

```typescript
// High-confidence facts
db.query("MATCH (s)-[:SUBJECT_OF]->(f:Fact)-[:OBJECT_IS]->(o) WHERE f.confidence > 0.9 RETURN s.name, f.predicate, o.name")

// Confidence-weighted PageRank
db.query("CALL algo.confidencePageRank()")

// Confidence-weighted shortest path
db.query("CALL algo.confidencePath()")
```

## Physical AI SDK — structured ingest

For physical AI deployments (robots, sensors, neural world models), the SDK provides typed ingest methods that set observation class, confidence, and provenance in one call, and optionally attribute the write to a named Agent.

```cypher
-- Write a sensor observation (class='observed')
-- _observation_class, _confidence, _source, _timestamp_ms set automatically
MATCH (e:Entity {id: 'player-07'})
SET e.x = 10.5, e.y = 7.2,
    e._observation_class = 'observed',
    e._confidence = 0.91,
    e._source = 'camera-3',
    e._agent_id = 'robot-01'

-- Write a neural WFM prediction (class='predicted')
MATCH (e:Entity {id: 'player-07'})
SET e.x = 11.0, e.y = 7.5,
    e._observation_class = 'predicted',
    e._confidence = 0.74,
    e._source = 'dreamer-v4',
    e._propagation_rule = 'neural-rollout'

-- Write a derived inference (class='inferred')
MATCH (e:Entity {id: 'player-07'})
SET e._observation_class = 'inferred',
    e._confidence = 0.85,
    e._propagation_rule = 'triangulation',
    e._source_count = 2
```

When `_agent_id` is set, the engine auto-creates `[:BELIEVES]` and `[:DETECTED]` edges linking the Agent to the Entity — enabling fleet-wide queries without any additional writes. See [Fleet Coordination](/guides/fleet-coordination) for the full multi-agent pattern.

```cypher
-- Fleet-wide: what does each agent believe about this entity?
MATCH (a:Agent)-[:BELIEVES]->(e:Entity {id: 'player-07'})
RETURN a.id, e._confidence, e._observation_class
ORDER BY e._confidence DESC
```

## Why this matters

- **Trust hierarchy** — observed > inferred > predicted. Make decisions at the right confidence level.
- **Debugging** — when a result is wrong, trace back to which observation or inference caused it.
- **Regulatory** — audit trails showing exactly how a conclusion was reached.
- **Agent safety** — LLM agents can distinguish hard facts from guesses before acting on them.

---

## When to use

- Any node or relationship that enters the world model through a sensor, API, or external data feed should be tagged `_observation_class: 'observed'` — this is the anchor class that downstream queries and skills treat as ground truth.
- Use `_observation_class: 'inferred'` when your application derives a new fact from existing observed nodes (e.g., a ReID algorithm matches two detections to the same person). The inferred class signals that downstream consumers should verify before high-stakes action.
- Use `_observation_class: 'predicted'` for any neural world model or statistical forecast output. This class explicitly communicates to query authors that the data is model-generated and should be treated as a planning input, not a confirmed fact.
- Use `CALL db.nodesByObservation.observed()` / `.inferred()` / `.predicted()` when you need to quickly enumerate all nodes of a given epistemic class — useful for monitoring dashboards and data quality sweeps.
- When setting `_agent_id` on a write, rely on the engine's automatic `:BELIEVES` and `:DETECTED` edge creation to build fleet-wide provenance graphs without additional manual mutations.

## When not to use

- Do not use observation class as a lifecycle status field (e.g., `predicted` → `observed` as a state machine for task completion). The three classes are epistemic, not workflow states. Use a separate `status` property for workflow state.
- Do not create custom observation class values beyond `observed`, `inferred`, and `predicted`. The built-in procedures (`db.nodesByObservation.*`) and confidence propagation algorithms only recognise these three strings. Custom values are silently ignored by the engine. <!-- DRAFT: confirm whether the engine enforces the closed set or silently ignores unknown classes -->
- Avoid setting `_observation_class: 'predicted'` on manually asserted facts (e.g., seed data you are certain about). The predicted class degrades trust in all downstream queries that filter on observation class — seed data should be `observed` at confidence 1.0.
- If your domain has no epistemic uncertainty (e.g., a simple CRUD app with no sensor or model inputs), adding `_observation_class` and `_confidence` to every node is unnecessary ceremony. The Evidence Model is designed for systems where data quality is heterogeneous.
- Do not rely on `CALL db.causalChain` for regulatory audit chains without first confirming the causal chain depth limit — very long derivation chains may be truncated. <!-- DRAFT: confirm causal chain depth limit and truncation behaviour -->

## Failure modes

- **Predicted nodes not cleaned up after confirmation**: a predicted contact (`_observation_class: 'predicted'`) is later confirmed by a direct sensor observation. If the application creates a new `observed` node rather than updating the existing predicted node, the graph accumulates duplicate entities — one predicted, one observed — pointing to the same physical object. Queries that don't filter on observation class see both.
- **Missing `_observation_class` on ingest**: a developer creates entities with `CREATE (p:Person {id: $id, x: $x, y: $y})` without setting `_observation_class`. The property is `null`. `CALL db.nodesByObservation.observed()` does not return these nodes. Downstream queries that assume all entities have an observation class will silently miss them.
- **Confidence propagation over inferred chains producing overconfident results**: an inference chain `A observed (0.9) → B inferred (0.85) → C inferred (0.88)` has a path confidence of ~0.67. If a downstream query filters at `> 0.6` and assumes the result is high quality, it may act on a fact that is actually marginally above the threshold.
- **`_agent_id` set without corresponding Agent node**: the engine auto-creates `:BELIEVES` and `:DETECTED` edges from Agent to Entity when `_agent_id` is set. If no `:Agent` node with that id exists in the graph, the auto-created edge references a non-existent source. Fleet-wide queries (`MATCH (a:Agent)-[:BELIEVES]->...`) return fewer results than expected. <!-- DRAFT: confirm whether the engine creates the Agent node automatically or requires it to pre-exist -->
- **`observed` class applied to model outputs for simplicity**: a developer uses `_observation_class: 'observed'` on neural WFM outputs because it produces cleaner query results (no need to add `AND _observation_class = 'observed'` everywhere). The epistemic distinction is lost. Safety-critical queries that rely on the class distinction to gate actions will not fire correctly.
