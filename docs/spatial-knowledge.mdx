# Spatial Knowledge Graph

**Exact spatial queries backed by a dynamic R\*-tree — no approximation, no separate spatial index service, no join between two systems.**

ArcFlow treats spatial coordinates as a first-class query dimension alongside graph relationships. K-nearest, radius, and bounding-box queries run against the R\*-tree index directly. Spatial filter and graph traversal compose in a single GQL statement — "find the 5 nearest warehouses that have inventory for SKU X" is one query, not two.

```cypher
-- Supply chain: spatial filter → graph traversal → property filter — one query
CALL algo.nearestNodes(order.location, 'Warehouse', 5)
  YIELD node AS wh, distance
MATCH (wh)-[:SUPPLIES]->(item:Item {sku: $sku})
WHERE item.stock > 0
RETURN wh.name, distance, item.stock
ORDER BY distance
```

## Why R*-tree, Not HNSW or Random-Walk

Most graph databases use the wrong spatial index:

| System | Index | Type | Exact? |
|---|---|---|---|
| Memgraph | USearch (HNSW) | Approximate ANN | No — tuned for embedding similarity |
| Neo4j GDS | Random-walk sampling | Approximate | No — controlled by `sampleRate`, `maxIterations` |
| PostGIS | GIST R-tree | Exact | Yes |
| **ArcFlow** | **R\*-tree** | **Exact** | **Yes** |

HNSW is optimised for high-dimensional semantic vectors — it finds "approximately similar" embeddings. For exact containment, bounding-box overlap, and dynamic 2D/3D boundaries it adds false positives and has no structural advantage. Random-walk sampling is explicitly approximate: it will miss answers.

The **R\*-tree** is the correct structure for exact 2D/3D geometric queries. Its key advantage over a basic R-tree: **forced reinsertion on node overflow**. Instead of immediately splitting a full node (which creates poor bounding boxes), R\*-tree removes ~30% of entries and reinserts them. This stochastic restructuring minimises both bounding-box overlap and dead space in MBRs — 2–5× faster queries than R-tree on clustered data.

## Spatial Primitives

### Point Types

```cypher
-- 2D cartesian (pitch, factory floor, map tile)
CREATE (p:Player {name: 'Alice', position: point({x: 34.2, y: 67.1})})

-- 3D cartesian (venue world space, robot arm, drone)
CREATE (r:Robot {name: 'AGV-7', position: point({x: 12.5, y: 8.3, z: 0.0})})

-- Geographic (WGS84)
CREATE (s:Sensor {name: 'Cam-01', location: point({latitude: 51.5074, longitude: -0.1278})})
```

Flat coordinate properties (`x`, `y`, `z`) are also indexed automatically.

### K-Nearest Neighbor

```cypher
CALL algo.nearestNodes(point({x: 10.0, y: 10.0}), 'Sensor', 5)
  YIELD node, distance
  RETURN node.name, distance
```

At 11K live entities: ≥ 2,000 queries/sec on CPU (R\*-tree path). The index is dynamic — inserts, updates, and deletes are O(log N) with no rebuild.

### Radius Search

```cypher
-- All players within 20 meters
MATCH (p:Player)
WHERE distance(p.position, point({x: 10.0, y: 10.0})) < 20.0
RETURN p.name, p.position
```

The query compiler detects `distance()` predicates and pushes them into the R\*-tree rather than scanning all nodes.

### Bounding Box

```cypher
-- Entities within a rectangular zone
MATCH (e:Entity)
WHERE e.position.x >= 0.0 AND e.position.x <= 50.0
  AND e.position.y >= 0.0 AND e.position.y <= 30.0
RETURN e.name
```

### Raycast

```cypher
-- What lies along a camera's line of sight?
CALL spatial.raycast(
  point({x: 0, y: 0, z: 2}),   -- origin
  point({x: 1, y: 0, z: 0}),   -- direction
  100.0                          -- max distance
) YIELD hit, distance
RETURN hit.name, distance
```

## Spatial + Graph in One Query

The reason spatial and graph coexist in one query is that the engine is in-process — the R\*-tree and the graph share unified memory. No serialization. No network hop between a spatial service and a graph database.

### Supply Chain
```cypher
-- Nearest warehouses with inventory → compute delivery via road graph
CALL algo.nearestNodes(order.location, 'Warehouse', 5)
  YIELD node AS wh, distance AS crow_flies
MATCH (wh)-[:SUPPLIES]->(item:Item {sku: $sku})
WHERE item.stock > 0
MATCH path = shortestPath((wh)-[:ROAD*]->(dest:Depot {id: $depot}))
RETURN wh.name, item.stock, length(path) AS road_hops
ORDER BY road_hops
```

### IoT / Fleet
```cypher
-- Vehicles within 500m, with valid license, maintained within 30 days
CALL algo.nearestNodes($zone_center, 'Vehicle', 50)
  YIELD node AS v, distance
WHERE distance < 500
MATCH (v)-[:DRIVEN_BY]->(d:Driver)-[:HOLDS]->(l:License)
WHERE l.expires > timestamp()
  AND v.last_service_date > timestamp() - 2592000000
RETURN v.plate, d.name, distance
```

### Live Production (50fps)
```cypher
-- Who's offside? Spatial filter → graph → rule check
CALL algo.nearestNodes(ball.position, 'Player', 22)
  YIELD node AS player, distance
MATCH (player)-[:PLAYS_FOR]->(team:Team {side: 'attacking'})
WHERE player.position.x > last_defender.position.x
RETURN player.name, player.position
```

## Coordinate Frame Validation

ArcFlow validates coordinate system metadata at ingest. Mixing coordinate frames — WGS84 positions in a Z-up cartesian graph — raises a **hard error** rather than silently producing wrong results. This is the "silent killer" class of spatial bug.

```cypher
CALL db.spatialMetadata()
  YIELD crs, meters_per_unit, up_axis, handedness, calibration_version
```

| Field | Default | Description |
|---|---|---|
| `crs` | `"cartesian"` | Coordinate reference system (`"cartesian"`, `"WGS84"`) |
| `meters_per_unit` | `1.0` | Scale — 1.0 for meters, 0.01 for centimeters |
| `up_axis` | `"Z"` | Z-up (venue/robotics world default) or Y-up (DCC/game engines) |
| `handedness` | `"right"` | `"right"` or `"left"` |
| `calibration_version` | `0` | Increment on venue re-calibration to invalidate stale queries |

The default Z-up matches OZ venue world convention (football pitch, arena, warehouse floor plan).

## Observability

```cypher
-- Which index path was taken for recent spatial queries?
CALL arcflow.spatial.dispatch_stats()
  YIELD query_id, index_path, candidates_filtered, result_count, latency_us
```

`index_path` will be one of: `rtree` (R\*-tree), `coarse_grid` (pre-filter only), or `scan` (fallback for small datasets below the dispatch threshold).

The coarse grid provides an O(1) pre-filter that reduces R\*-tree candidate sets by ~95% on typical datasets (e.g. 8m cells over a 100m × 70m pitch with 50 entities → 156 cells, ~5 entities/cell).

## Multi-Clock Sensor Fusion

Different sensors report at different rates. ArcFlow's temporal clock system handles this natively:

```cypher
-- Register sensor clocks
CALL db.clock('camera_01', 'sensor', 33)   -- 30fps
CALL db.clock('lidar_01', 'sensor', 100)   -- 10Hz
CALL db.clock('gps_01', 'sensor', 1000)    -- 1Hz

-- Query fused state across clock domains
MATCH (r:Robot) RETURN r.position, r.orientation, r._clock_domain
```

## Use Cases

### Digital Twin (Venue / Building / Factory)

```cypher
CREATE (v:Venue {name: 'Stadium'})
CREATE (c:Camera {name: 'Cam-01', position: point({x: 0, y: 50, z: 15})})
CREATE (z:Zone {name: 'Pitch'})

-- Real-time tracking
MATCH (p:Player {jersey: 7})
SET p.position = point({x: 34.2, y: 67.1, z: 0.0})

-- Spatial query: who's near the ball?
CALL algo.nearestNodes(ball.position, 'Player', 5)
  YIELD node, distance
RETURN node.name, distance
```

### Warehouse Robotics

```cypher
-- Assign nearest idle robot to a pick location
MATCH (r:Robot) WHERE r.status = 'idle'
CALL algo.nearestNodes($pick_location, 'Robot', 1)
  YIELD node AS nearest
SET nearest.task = 'pick', nearest.target = $pick_location

-- Collision avoidance
MATCH (a:Robot), (b:Robot)
WHERE a <> b AND distance(a.position, b.position) < 2.0
RETURN a.name, b.name, distance(a.position, b.position)
```

### Epidemiology

```cypher
-- People within 10m of patient zero → BFS contacts → temporal filter
CALL algo.nearestNodes(patient_zero.location, 'Person', 1000)
  YIELD node AS contact, distance
WHERE distance < 10.0
MATCH path = (patient_zero)-[:CONTACT*1..3]->(contact)
WHERE ALL(r IN relationships(path) WHERE r.timestamp > timestamp() - 172800000)
RETURN contact.name, length(path) AS degrees, distance
```

## Compared to Other Approaches

### PostGIS / Relational Spatial Extensions
PostGIS has richer geometry types (LineString, Polygon, MultiPolygon) and mature GIS tooling. ArcFlow's advantage: spatial predicates compose directly with graph traversal — no join between a PostGIS table and a graph database. If you need complex polygon operations without graph structure, PostGIS is the better tool. If you need "find entities near X, then traverse their relationships," ArcFlow avoids the two-system join.

### Memgraph / Neo4j Spatial
Memgraph uses HNSW (approximate ANN — tuned for embeddings, not exact geometry). Neo4j GDS uses random-walk sampling (approximate, misses answers). ArcFlow uses R\*-tree (exact). For containment queries, bounding-box overlap, and correctness-critical applications (offside detection, collision avoidance, delivery routing), approximation is not acceptable.

### cuSpatial / GPU-native Spatial
cuSpatial uses a quadtree, which has higher false-positive rates for complex overlapping geometries than R\*-tree and cannot be uploaded to GPU memory as a flat array. ArcFlow uses cuSpatial in Phase 4 for analytics-class workloads (trajectory analytics, point-in-polygon at millions of entities) while keeping the R\*-tree as the primary query index. The GPU index (Phase 4) is a custom FlatBVH — contiguous array, GPU-uploadable, wgpu compute shader.

### ROS2 / Nav2
ArcFlow complements Nav2: Nav2 handles real-time path planning with hard latency guarantees; ArcFlow provides the persistent spatial knowledge layer — mission planning, post-incident replay, spatial-temporal reasoning. The multi-clock domain system maps directly to ROS2's multi-rate sensor model.
