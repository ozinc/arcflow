# GPU Acceleration

ArcFlow runs graph algorithms as **parallel operations across the entire graph at once** — not as sequential edge traversals. Most graph databases walk the graph one node at a time: visit a node, follow an edge, visit the next. That is sequential, cache-hostile, and does not map to GPU hardware.

ArcFlow's graph engine represents the world model as a compact parallel structure and processes algorithms across all nodes simultaneously. PageRank, BFS, connected components, community detection — each runs as a single parallel pass, not a recursive walk. This maps directly to GPU thread blocks.

The developer writes one query. The engine picks the fastest path.

```cypher
CALL algo.pageRank()
```

---

## Automatic Hardware Dispatch

ArcFlow detects available hardware at startup and routes each operation to the fastest available backend:

1. **Small graphs (< 200 nodes)** — CPU (dispatch overhead exceeds GPU benefit at this scale)
2. **Apple Silicon (macOS / iOS)** — GPU compute via Metal. Unified memory means no CPU→GPU copy overhead.
3. **NVIDIA GPU (Linux / Windows)** — GPU compute via CUDA kernels. Dynamic driver loading — no compile-time GPU dependency.
4. **CPU fallback** — ArcFlow's own parallel CPU implementation when no GPU is present.

Zero configuration. The same GQL query runs on a laptop, a workstation, or a GPU cluster.

## Benchmark Results

Performance measured against ArcFlow's own CPU path on the same hardware:

| Algorithm | CPU | GPU Speedup |
|---|---|---|
| PageRank | 154M nodes/sec | 2.4x |
| BFS Frontier | 6.3M edges/sec | 3.5x |
| Vector Distance | 25K queries/sec | 4.2x |
| Triangle Count | 943K nodes/sec | 19.8x |
| Community Detection | 185K nodes/sec | 29.6x |

Speedup is most pronounced for algorithms with high parallelism (community detection, triangle counting). For simpler traversals, the CPU path is already fast — GPU dispatch adds value when the graph is large and the algorithm is inherently parallel.

## GPU Spatial Queries

Spatial queries follow a separate dispatch path from graph algorithms. The cost model accounts for query shape, candidate count, and GPU transfer overhead:

| Lane | When | Typical use |
|---|---|---|
| `CpuLive` | ≤ 500 candidates | LIVE queries, real-time tracking |
| `CpuBatch` | > 500 candidates (CPU faster than transfer) | Analytics, replay |
| `GpuLocal` | > 50K candidates, fits single GPU | High-density spatial |
| `GpuMulti` | Exceeds single GPU memory | Stadium-scale entity tracking |

Routing is cost-based — kernel launch overhead and memory bandwidth are measured per device and factored in. There are no hardcoded size thresholds.

### Multi-GPU Spatial Partitioning

Spatial data is partitioned across GPU devices by a stable hash of `node_id`. Queries spanning partition boundaries fan out to all relevant GPUs and merge results. Devices connected via NVLink, NVSwitch, or Infinity Fabric form peer islands — within an island, work-stealing happens without PCIe transfer cost.

```cypher
-- Same spatial query — routes to GpuMulti automatically when warranted
CALL algo.nearestNodes($center, 'Entity', 100) YIELD node, distance RETURN node.name
```

## Metal GPU (Apple Silicon)

On macOS and iOS, ArcFlow dispatches graph and spatial operations to Metal compute shaders. Apple Silicon's Unified Memory Architecture means the graph representation does not need to be copied to the GPU — CPU and GPU share the same physical memory.

```
PageRank on 10K nodes — CPU: 0.6ms / GPU: 0.25ms (2.4x)
```

Metal dispatch is automatic. If Metal is available and the graph exceeds the minimum dispatch threshold, it routes to GPU. If Metal fails or is unavailable, it falls back to CPU transparently.

---

## CUDA GPU (Linux / Windows)

On systems with NVIDIA GPUs, ArcFlow dispatches to CUDA via dynamic loading. No compile-time CUDA dependency — the driver is discovered at runtime. Covering PageRank, BFS, vector distance, triangle counting, community detection, and spatial operations across different workload scales.

All routing is automatic. The developer writes one query; the engine selects the appropriate backend based on hardware and data size.

## GPU Introspection

### `CALL db.gpuStatus()`

Returns one row per CUDA device in the pool. Use this to check device availability and current load.

```cypher
CALL db.gpuStatus() YIELD device_id, inflight, sm_count, vram_mib, status
```

| Column | Type | Description |
|---|---|---|
| `device_id` | int | CUDA device index |
| `inflight` | int | Currently executing GPU kernels |
| `sm_count` | int | Streaming multiprocessor count |
| `vram_mib` | int | Device VRAM in MiB |
| `status` | string | `"available"` (inflight < 8) or `"saturated"` |

Returns `{device_id: "N/A", status: "no CUDA devices"}` when no CUDA hardware is present.

### `CALL dbms.gpuThresholds()`

Returns the GPU dispatch registry — the minimum requirements for each algorithm. Use this to understand why an algorithm is or isn't routing to GPU.

```cypher
CALL dbms.gpuThresholds()
  YIELD algorithm, min_input_size, bytes_per_element, validated, cuda_min_cc
```

| Column | Description |
|---|---|
| `algorithm` | Algorithm name (e.g. `"pageRank"`, `"leiden"`) |
| `min_input_size` | Minimum node count before GPU dispatch is considered |
| `bytes_per_element` | Per-node memory estimate for transfer cost calculation |
| `validated` | Whether the kernel has been validated on this hardware |
| `cuda_min_cc` | Minimum CUDA compute capability required (e.g. `"9.0"`) or `"none"` |

### `CALL arcflow.spatial.dispatch_stats()`

Observability for spatial query routing decisions.

```cypher
CALL arcflow.spatial.dispatch_stats()
  YIELD lane_chosen, estimated_candidates, actual_candidates,
        prefilter_us, rtree_us, gpu_transfer_us, kernel_us, total_us
```

`gpu_transfer_us` and `kernel_us` are non-zero only when a GPU lane was chosen. Use these to validate that the cost model is routing as expected for your workload.
