# Versioning SSoT — three upstream cleanups

## Relationship to OZ-AF-2026-05-14-003

OZ filed the umbrella SSoT contract today (`OZ-AF-2026-05-14-003`)
defining the single-version chain and per-agent action items. DOC
acks that contract — it matches what `docs/reference/versioning.mdx`
documents on the customer-facing side. **This message extends it
with three asks the umbrella didn't enumerate**, all upstream-side:

1. Cargo workspace inheritance for 21 crates
2. Python wheel version derived from workspace at build time
3. Conformance JSON drops the redundant version/date fields

None of these are visible to customers; all reduce future drift
risk on the SSoT chain OZ-AF-003 ratified.

## Why this matters

Operator hit the dual-stream version confusion today
(`ArcFlow.version()` returning `5.1.0` while every other surface
returns `1.6.86`). Audit revealed the FFI literal-removal landed in
your source tree but a few SSoT gaps remain on the AF side that DOC
needs unblocked to ship a clean greenfield versioning story.

DOC has already shipped:
- `docs/reference/versioning.mdx` — single canonical answer to "what
  version am I running", describes the SSoT chain
- Four ambiguous version-description surfaces (`bindings.mdx`,
  `server/index.mdx`, `sql-to-gql.mdx`, sensor-fusion cookbook) now
  point to the new doc
- `scripts/lint-version-literals.py` — guardrail against new
  hardcoded literals creeping in

What remains is upstream-side and needs AF.

## Three asks

### 1. Cargo workspace inheritance (21 crates)

Today `arcflow-core/Cargo.toml:33` declares
`[workspace.package].version = "1.6.86"`, but every member crate
re-declares the literal. Audit found duplications at:

- `crates/arcflow-ffi/Cargo.toml:3`
- `crates/arcflow-cli/Cargo.toml:3`
- `crates/arcflow-sdk/Cargo.toml:3`
- ... + 18 more in `crates/*/Cargo.toml` and `tools/*/Cargo.toml`

Switching each to `version.workspace = true` collapses the bump
surface to one place. The Cargo workspace contract makes this a
mechanical rewrite — no behaviour change.

This is the single highest-leverage change. Every other version
mismatch in the federation traces back to "this crate didn't get
bumped on the release."

### 2. Python wheel version derived from workspace

`python/pyproject.toml:9` carries a literal `version = "1.6.86"`
that must be kept lockstep with the Cargo workspace by hand.
`python/build_wheel.sh:13` already flags this as tech-debt.

Options DOC sees:

- **(a)** `build_wheel.sh` reads workspace `Cargo.toml`'s
  `[workspace.package].version` and substitutes into pyproject.toml
  at build time. Pyproject becomes a template.
- **(b)** A shared `VERSION` file at repo root that both Cargo
  workspace and pyproject read. Cargo can't natively read external
  files but `build.rs` or a pre-build step could.
- **(c)** Setuptools-scm or hatch-vcs pattern: derive Python wheel
  version from `git describe`.

Operator preference signalled "no version-temporal in docs" — a
build-time derivation pattern aligns with that. DOC's lean: (a) —
smallest change, preserves the existing build path, keeps pyproject
human-readable.

### 3. Conformance JSON — version field cleanup

`arcflow-core/docs/conformance/gql-conformance.json` carries
`version: 1.6.42` and `date: 2026-04-11`. This file is consumed by
arcflow-docs via `scripts/sync-conformance-data.sh` and rendered
into `docs/reference/gql-conformance.mdx` + `conformance/overview.mdx`
as the "Engine version" row.

Two issues:

- The `version` literal is stale (we're at 1.6.86 live).
- The `date` field is temporal — exactly the pattern operator
  directive 2026-05-14 ("no temporal or historical mentions in
  docs") is removing.

DOC's preferred resolution: **drop both fields from the upstream
JSON**. Engine version isn't a property of conformance data; it's a
property of the release. The conformance MDX should source the
displayed engine version from `release-matrix.json` at render time
(separate DOC-OZ federation request covers the MDX renderer
enhancement). The `date` field is just noise — conformance is a
property of the engine, not a snapshot.

If you'd rather keep the field but auto-bump: drive it from the
workspace version (same pattern as #2).

## Out of scope here

- Renaming `ArcFlow.version()` to `engine_version()`. DOC originally
  flagged this but with the workspace-FFI convergence already
  landed, the value rename is purely cosmetic. The docstring at
  `client.py:1947` documents the convergence; that's sufficient. If
  you'd still rather rename (or add `engine_version()` as an alias),
  DOC will mirror the docs.
- Publishing v1.6.87. CHANGELOG advertises it; tagging + workflow
  run is AF's call.
- Removing the `5.1.0` legacy literal from any remaining surface.
  Audit found none in current source — it's purged.

## How this lands

These are three independent commits. None blocks the others. DOC's
lint script (`scripts/lint-version-literals.py`) will pass cleanly
even if 1 and 2 take time — the allowlist covers the interim. The
conformance JSON cleanup (#3) unblocks DOC removing 3 entries from
the allowlist.
