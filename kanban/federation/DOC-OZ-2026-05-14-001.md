# Versioning SSoT — render `{{ENGINE_VERSION}}` from manifest at MDX build time

## Why this matters

Operator's directive 2026-05-14: "fix our versioning mess. we need
clarity, and SSoT management on it" + "no need for temporal or
historical mentions" + "Full SSoT: render version from manifest."

DOC has shipped the customer-facing answer
(`docs/reference/versioning.mdx`) and the guardrail
(`scripts/lint-version-literals.py`). What's left is the
build-time substitution that lets us strip the remaining hardcoded
literals from prose.

OZ already does the right thing for the **install matrix UI** — it
reads `engine.version` from `release-matrix.json` at build time via
`apps/cloud/website/src/lib/release-matrix.ts`. This request
extends that pattern to the **MDX docs renderer**.

## What's being asked

A new placeholder convention in the MDX docs build:

```mdx
The current ArcFlow version is `{{ENGINE_VERSION}}`. Install with:

    ARCFLOW_VERSION={{ENGINE_VERSION}} curl -fsSL https://oz.com/install/arcflow | sh
```

renders as (with manifest fetched at build):

```html
<p>The current ArcFlow version is <code>1.6.86</code>. Install with:</p>
<pre><code>ARCFLOW_VERSION=1.6.86 curl -fsSL https://oz.com/install/arcflow | sh</code></pre>
```

### Implementation shape (OZ to design)

Likely an MDX remark/rehype plugin that:

1. Loads `release-matrix.json` at build start (cached in-process).
2. Walks each MDX file, substituting `{{ENGINE_VERSION}}` (and
   ideally `{{ENGINE_VERSION_RAW}}` without surrounding code-fence
   considerations) with the manifest value.
3. Optionally extends to `{{ENGINE_VERSION_MINOR}}` / similar shapes
   if needed — DOC's initial use case is just the full version.

Build-time, not runtime — so the rendered HTML is static, no
client-side fetch needed.

### Source of truth

`https://github.com/ozinc/arcflow/releases/latest/download/release-matrix.json`
→ `.engine.version`

The same URL OZ already reads for the install-matrix UI. Single
SSoT for the whole frontend.

### Files DOC will convert to placeholders once shipped

DOC currently maintains 8+ files with hardcoded version literals in
prose (see `scripts/lint-version-literals.py` allowlist). After
the placeholder lands, conversion is mechanical: each literal
`1.6.86` becomes `{{ENGINE_VERSION}}`. The allowlist shrinks back
to just the build-time consumers (cookbook `pyproject.toml`,
`install.sh` env-var example) and the SoT file itself
(`docs/reference/data/SYNC.json`).

Cookbook README "Pins:" lines (11 files) are the biggest win — each
release currently requires bumping 11 files by hand.

## What's NOT being asked

- Runtime version display (e.g. a footer that shows the currently-
  resolved engine version on every page). Build-time is sufficient
  and avoids the client-fetch surface.
- A separate placeholder convention for `installation.mdx` install
  commands — `{{ENGINE_VERSION}}` handles all cases.
- Any change to cookbook `pyproject.toml` or `meta.toml` rendering —
  those are consumed by `uv` / `pip` directly, not the MDX renderer.

## Companion federation request

`DOC-AF-2026-05-14-002` asks AF to drop the redundant `version` and
`date` fields from `arcflow-core/docs/conformance/gql-conformance.json`
so the conformance dashboard MDX can use `{{ENGINE_VERSION}}`
instead of a hand-bumped literal. The two requests together fully
close the SSoT loop for docs prose.
