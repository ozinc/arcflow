# Two release-publication polish items — manifest tag field + explicit --latest

## Why this exists

End-to-end audit of the v0.7.1 publish chain (in response to operator's
"do full publish" + "is `arcflow@latest` naming a concern?" review)
verified the GitHub Releases `/latest/download/<filename>` pattern is
working correctly. Two small surface refinements surfaced that don't
break anything today but are worth tightening before v0.7.2:

## (1) `engine.tag` is empty in release-matrix.json

Manifest at `https://github.com/ozinc/arcflow/releases/latest/download/release-matrix.json`
currently renders:

```json
{
  "engine": {
    "version": "0.7.1",
    "tag": "?"
  },
  ...
}
```

`engine.version` works fine (the `"auto"` sentinel in RELEASE-MATRIX.toml
resolves correctly). `engine.tag` is empty. The render script at
`scripts/render-release-matrix-json.py` presumably wants both — version
for display, tag for URL-construction — but only populates the first.

Impact: a consumer reading the manifest who wants to construct
`https://github.com/ozinc/arcflow/releases/download/<tag>/<asset>`
URLs (the canonical permalink form, vs the `/latest/download/` redirect)
has to prefix `v` themselves from the version string. Minor friction;
also a SemVer pre-release suffix (`v0.7.2-rc.1`) would have a tag like
`v0.7.2-rc.1` while the version is `0.7.2-rc.1` — same string today but
divergent in the future if tag prefixes change.

Fix: have `render-release-matrix-json.py` populate `engine.tag` from
the same `git describe --tags --abbrev=0` it uses for the version. Just
include the leading `v`.

## (2) `gh release create` doesn't pass `--latest` explicitly

`release-binaries.yml:683`:

```yaml
gh release create "${TAG}" \
  --repo "${REPO}" \
  --title "ArcFlow ${TAG}" \
  --notes "${NOTES}" \
  release-assets/*
```

The v0.7.1 release IS correctly marked as latest (`gh api releases/latest`
returns `tag_name: v0.7.1`, `prerelease: false`, `draft: false`). But that's
the gh CLI's default behavior for the most recent release — implicit. A
future workflow edit could regress this silently (e.g. someone adds
`--prerelease` for `-rc.N` cuts and forgets the inverse path).

Fix: pass `--latest` explicitly on the production release branch:

```yaml
gh release create "${TAG}" \
  --repo "${REPO}" \
  --latest \
  --title "ArcFlow ${TAG}" \
  --notes "${NOTES}" \
  release-assets/*
```

For `-rc.N` pre-releases, pass `--prerelease` instead. The branching
logic can key off whether `TAG` matches `*-rc.*`.

## Why low-severity

Neither item breaks anything today. Both are pre-emptive guard rails. DOC
filed them because the end-to-end audit landed them on the "could
silently regress" pile, and v0.7.2 is staged on main but un-tagged — a
clean opportunity to bake these in before the next tag-cut.

## What DOC needs from AF

A small commit on arcflow-core main applying both. v0.7.2's tag-cut
(whenever the operator triggers it) then ships with both improvements
baked in.

## Resolution

Close by linking the AF commit + the next release whose `release-matrix.json`
shows `engine.tag = "vX.Y.Z"`. DOC verifies and mirrors the closure.
