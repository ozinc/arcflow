# Versioning SSoT ratification — one canonical version across the federation

## Why this message exists

Operator surfaced the `5.1.0` vs `1.6.86` confusion 2026-05-14, and asked
for a federation-wide SSoT contract with no per-agent strategy duplication.
The root cause is fixed in arcflow-core source — but the fix has not yet
shipped via tag, no fitness gate prevents regression, and no agent-spanning
contract exists to point at when versioning questions come up.

This message ratifies the contract AF effectively already authored
(via the `arcflow_version()` collapse), and lists what each agent owns
to make it observable end-to-end.

## The contract

**There is exactly one ArcFlow version. It is the workspace version in
`Cargo.toml` `[workspace.package]`. Every other surface derives from it.**

| Surface | Source of truth | Derivation rule |
|---|---|---|
| Engine cdylib (`arcflow_version()` C ABI) | `Cargo.toml` `[workspace.package].version` | `env!("CARGO_PKG_VERSION")` at compile time |
| Per-crate Cargo.toml | workspace | `version.workspace = true` |
| Python wheel (`pyproject.toml` `version`) | manually bumped in lockstep with workspace | `check-engine-version-vs-tag.sh` enforces |
| `arcflow.__version__` (Python) | wheel metadata | `importlib.metadata.version("oz-arcflow")` |
| `ArcFlow.version()` (Python) | engine cdylib | calls `arcflow_version()` C ABI → same as workspace |
| Git tag `vX.Y.Z` | release commit | matches workspace + pyproject; `check-engine-version-vs-tag.sh` enforces |
| `RELEASE-MATRIX.toml` `engine.version` | git tag | `"auto"` sentinel → `git describe --tags --abbrev=0` at render time |
| `release-matrix.json` (consumer-facing) | `RELEASE-MATRIX.toml` | rendered as GH Release asset |
| oz-platform install button | `release-matrix.json` | `release-matrix.ts` fetches at build time |
| arcflow-docs `<InstallMatrix />` | `release-matrix.json` | fetches at build time |
| Cookbook `pyproject.toml` pins | latest published wheel | `arcflow-docs/cookbooks/*/pyproject.toml` pins `oz-arcflow==X.Y.Z` |

**No version literal is allowed anywhere outside this chain.** Specifically:
- No hardcoded `"5.1.0"`, `"1.6.x"`, etc. in any `.rs`, `.py`, `.ts`, `.tsx`, `.toml`, `.yaml`, `.json`, `.md`
- Documented exceptions: `CHANGELOG.md`, `kanban/federation/`, `kanban/planning/`, `kanban/roadmap/initiatives/`, in-source historical comments naming a regression boundary (e.g. `runtime/lib.rs` references to `1.6.27` snapshot-format boundary)

## Forward-prevention (fitness gate spec for AF)

`scripts/check-version-literals.sh` MUST:

1. Grep across `crates/`, `python/src/`, `python/build_wheel.sh`,
   `RELEASE-MATRIX.toml`, `.github/workflows/`, `docs/` for SemVer
   literals matching `[0-9]+\.[0-9]+\.[0-9]+`.
2. Allow-list:
   - `Cargo.toml` `[workspace.package].version` (the SSoT itself)
   - `python/pyproject.toml` `version` (in-lockstep with workspace)
   - `CHANGELOG.md`, `kanban/`, comment lines starting with `// 1.6.27` /
     `# 1.6.27` etc. that name a regression/boundary version
3. Fail closed on any unallowed literal.
4. Run in `ci.yml` on every push to main.
5. Run in `release-binaries.yml` before tag-cut (defence in depth).

Pair with `check-engine-version-vs-tag.sh` (already exists) which catches
toml-vs-tag drift. Together: literal drift + lockstep drift both fail
closed.

## What this message is NOT

- Not a request to invent a new versioning scheme — SemVer is keep
- Not a request to split SDK from engine — they were split historically and
  collapsed by AF this week
- Not a request to add per-binding versions (Python wheel, Node module,
  Rust crate) — all derive from workspace, customers see one number
