# Audit + cleanup pass: arcflow-docs side of the R2 → GH Releases cutover

across all three repos. The install chain is wired end-to-end and fails
cleanly until the operator cuts `v1.6.27`. While we wait for the
tag-cut, the public docs surface (this repo) still needs an audit for
stale R2 references and aspirational install commands the cutover may
have left behind.

## What's being asked

Eight targeted sweeps. None requires waiting for the tag-cut; all are
addressable now.

### 1. R2 binary URL sweep

    grep -rEn "pub-a0a196dbe10340f8af22524547fdd476|staging\.oz\.com/pypi" . \
      | grep -vE "node_modules/|\.git/"

For each hit: if it's a binary-distribution surface (install command,
release tarball, version pin) — remove or rewrite to the GH Releases
shape. If it's a marketing asset (blog image, hero video, team avatar)
— leave alone; R2 still hosts marketing media legitimately.

four Vercel rewrites on oz-platform were deleted in `bbbd5452`). Any
`pip install --index-url https://staging.oz.com/pypi/simple/` line must
become either a planned-status reference or be deleted.

### 2. Stale install-command sweep

    grep -rEn 'pip install (-|--index-url|arcflow|oz-arcflow)|npm install arcflow|cargo (add|install) arcflow|docker run.*arcflow|install_arcflow' . \
      | grep -vE "node_modules/|\.git/"

    curl -fsSL https://staging.oz.com/install/arcflow | sh

2026-Q4) or refused (docker, by design). Bare `pip install oz-arcflow`
without a "coming in RAM-C2" qualifier is wrong. Manifest-driven render
surfaces (the install matrix MDX block) get the qualifier from the
manifest; don't restate it in prose.
