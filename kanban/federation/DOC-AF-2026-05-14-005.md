# Metal introspection gap — Mac binary IS Metal-enabled but `db.gpuStatus` only reports CUDA

## Summary

DOC verified end-to-end that the shipped Mac libarcflow.dylib has Metal fully
compiled in (linker dep on Metal.framework, MSL shader source baked in,
arcflow-ffi has no feature-gate so Metal is unconditional, runtime calls
`MTLCreateSystemDefaultDevice()` at `metal/mod.rs:488`). Operator asked "are
we sure we're compiling the Mac version with full Mac GPU support?" — answer
is **yes**, with hard evidence.

**However**, two customer-facing introspection surfaces give the *opposite*
impression on Apple Silicon:

1. **`CALL db.gpuStatus()`** returns:
   ```
   device_id=N/A | inflight=0 | sm_count=0 | vram_mib=0 | status="no CUDA devices"
   ```
   That's because the implementation at
   `crates/arcflow-runtime/src/lib.rs:10260` enumerates CUDA only — there's
   no Metal-device branch. On a Mac, a customer running this procedure walks
   away thinking they have no GPU.

2. **`CALL db.capabilities()`** reports `gpu_backend | cpu` on a small/empty
   graph because Adaptive Dispatch keeps tiny workloads on CPU below the
   dispatch threshold. Same field name, two different meanings: "Metal
   compiled in" (always true on Mac) vs "currently dispatched backend"
   (workload-dependent). Conflated under one field, this reads as
   "Metal unavailable" to anyone running an exploratory query.

## Evidence Metal IS compiled in

| Layer | Evidence |
|---|---|
| Linker | `otool -L libarcflow.dylib` shows `Metal.framework`, `Foundation.framework`, `Accelerate.framework`, `libobjc.A.dylib` |
| Shader source | `strings libarcflow.dylib | grep simdgroup` reveals `simdgroup_matrix<float, 8, 8>`, `simdgroup_load`, `simdgroup_multiply_accumulate`, `simd_sum`, "Pass 1: within-simdgroup inclusive prefix sum. Apple8+" — the M4-tier kernels are embedded as MSL |
| Source crate | `arcflow-ffi/Cargo.toml` has no `[features]` section — Metal is unconditional |
| Source dir | `arcflow-runtime/src/metal/` has 10 modules (binary_archive, counters, dispatch, graphblas, heap, mod, msl, pagerank, query_exec, residency) |
| Runtime probe | `MTLCreateSystemDefaultDevice()` called at `metal/mod.rs:488` |
| Capability self-report | `gpu_spmv_semirings | 8`, `gpu_spgemm | hash_table`, `gpu_deterministic_f64 | true` — values that would be 0 / false / absent if Metal weren't probed |

## What the operator sees today (verbatim transcript)

```
$ arcflow query 'CALL db.gpuStatus() YIELD device_id, inflight, sm_count, vram_mib, status'
device_id | inflight | sm_count | vram_mib | status
N/A       | 0        | 0        | 0        | no CUDA devices

$ arcflow query 'CALL db.capabilities() YIELD capability, value WHERE capability =~ "gpu.*"'
gpu_backend          | cpu
gpu_devices          | 0
gpu_spgemm           | hash_table
gpu_spmv_semirings   | 8
gpu_deterministic_f64 | true
cugraph_probe        | cuGraph not available — libcugraph_c.so not found or ARCFLOW_NO_GPU=1
```

A customer or coding agent reading this output concludes "no GPU" — even
though every kernel needed for Metal-accelerated PageRank, BFS, SpMV,
SpGeMM is present in the binary.

## Suggested shape (AF's call)

**(a) Extend `db.gpuStatus` to enumerate Metal on macOS** (preferred — same
procedure name, fewer surface things to document):

```
device_id | family    | kernels_available | inflight | unified_memory_mib | status
0         | Apple9    | 27                | 0        | 24576              | available
```

Single row on Apple Silicon (always 1 device — the integrated GPU). On
Windows/Linux-no-CUDA: zero rows (or one row with `status="no GPU detected"`).
On Linux-with-CUDA: one row per CUDA device. The `family` column distinguishes
backends (`Apple7`, `Apple8`, `Apple9`, `Apple-M3+`, or a CUDA SM string).

**(b) Add a sibling `db.metalStatus` procedure** that reports Metal device
info specifically. `db.gpuStatus` continues to enumerate CUDA only.
`db.capabilities()` gains a `metal_available` boolean to disambiguate.

**(c) Just relabel `db.capabilities()`** — add a `metal_compiled_in` field
and rename `gpu_backend` to `current_dispatch_backend` to clear the
conflation. Don't touch `db.gpuStatus`.

DOC's lean: **(a)**. Same name, one procedure, no doc-fork — the
right contract is "what GPU are you running on" not "what CUDA devices
exist."

## Why this matters now

- **DOC's `docs/gpu.mdx`** documents Metal as a live capability (the
  per-family kernel-selection table I shipped in commit `0c56de9` lists
  `simd_sum` on Apple8+, `atomic_float` on Apple9+, `simdgroup_matrix` on
  M3+). If a Mac customer runs `db.gpuStatus()` to verify the docs and sees
  "no CUDA devices," they'll file an issue assuming the docs are wrong.
- **Coding agents** parse `gpu_backend = cpu` as definitive and recommend
  alternatives (DuckDB, etc.) based on the wrong signal.
- **The MRL broadcast `MRL-broadcast-2026-05-14-v0-7-1-alpha-convention`**
  asked consumers to flag any consumer-visible breakage from the v0.7.1 cut.
  This isn't breakage from v0.7.1 specifically; it's a pre-existing
  inspection gap that v0.7.1 didn't fix. Logged here so it doesn't get lost.
