## What this means for project-merlin

### `versions.py` and the `sdk_version` / `ffi_version` split

Your module-level docstring at `src/project_merlin/versions.py:1-25`
is the cleanest downstream account of the historical split. After
the FFI fix lands in published wheels, both values converge. Two
choices:

**(a) Keep the split surface (defensive).** `sdk_version` from
`importlib.metadata`, `ffi_version` from `ArcFlow.version()`. They'll
return the same string on rebuilt wheels but the API contract
remains "these can in principle differ; we expose both." This is
honest and survives any future regression.

**(b) Simplify to one `version` field.** After confirming
convergence on a rebuilt wheel, collapse `versions.py` to a single
field. Cleaner customer-facing API. But loses the diagnostic-split
property.

DOC's lean: **(a)**. The cost is low (one extra dict field) and the
diagnostic property pays off the first time a wheel ships with a
mismatched FFI version. The greenfield-no-temporal rule doesn't
apply here — `versions.py` is *internal*, not customer-facing docs.

### Hardcoded `EXPECTED_SDK`

`tools/probe_tier1.py:388` and `versions.py:35,62` carry
`EXPECTED_SDK = "1.6.86"` / `expected: str = "1.6.86"`. These will
break when 1.6.87 (or whatever the next AF release is) ships.

Suggested fix:

```python
from importlib.metadata import version
EXPECTED_SDK = version("oz-arcflow")  # the wheel that's installed
```

This makes the assertion "the FFI binding's reported version
matches the installed wheel's version", which is the real
invariant. Drift between installed wheel and reported FFI version
is the bug; the specific version number is incidental.

Same pattern for `assert_sdk_matches_wheel(expected: str = "1.6.86")` —
default to `version("oz-arcflow")`.
