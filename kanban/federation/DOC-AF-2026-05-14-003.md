# Strip "reactive" framing from synced conformance reports

## Why this matters

Operator directive 2026-05-14: "we do never want to use the word or
term Reactive." Repository-wide sweep cleaned arcflow-docs except
for two surfaces that are synced from arcflow-core via
`scripts/sync-conformance-data.sh`:

- `arcflow-core/conformance/reports/extensions.md` → arcflow-docs
  `docs/reference/data/extension-regression.md`
- `arcflow-core/docs/conformance/arcflow-extensions-catalog.md` →
  arcflow-docs `docs/reference/data/arcflow-extensions-catalog.md`

Both currently carry:

> `CREATE TRIGGER` — OK (legacy `CREATE REACTIVE SKILL` syntax retained
> for backward compatibility per PAT-0037 / PAT-0038)

DOC stripped these locally (commit linked below), but they revert the
next time `sync-conformance-data.sh` runs. The fix needs to land
upstream.

## The ask

Strip the "legacy `CREATE REACTIVE SKILL`" mentions from:

1. `arcflow-core/conformance/reports/extensions.md` (line containing
   the `CREATE TRIGGER` entry)
2. `arcflow-core/docs/conformance/arcflow-extensions-catalog.md` (the
   `## Triggers (PAT-0037, I-INIT-0035)` section)

Replace with:

```
- [PASS] **CREATE TRIGGER** (10ms) — OK
```

and

```
**Syntax:**
\`\`\`gql
CREATE TRIGGER auto_tag ON :Article WHEN CREATED RUN SKILL tag_skill
\`\`\`

**GQL compatibility:** No GQL equivalent.
**Semantics:** Event-driven skill execution on CDC events. Skills fire when graph conditions match.
**Evidence:** I-INIT-0035, PAT-0037
```

i.e. drop the "legacy backward compatibility" paragraph entirely.
The greenfield rule says: describe only the current syntax. If the
old `CREATE REACTIVE SKILL` keyword still tokenizes in the engine
grammar, that's an engine-internal detail; the public docs only show
the current name.

## Two implementation paths

**(a) Cleanest — retire the legacy keyword in the engine grammar.**
Remove the `REACTIVE SKILL` parser alias. New code uses `TRIGGER`
exclusively. PAT-0038 framing already says "user-facing keyword
'reactive' is retired"; this just finishes the retirement.

**(b) Lightest — just strip the back-compat prose.** Leave the
parser alias intact (it harms nothing if undocumented). The synced
conformance reports stop mentioning it. Customers searching the
docs find only `CREATE TRIGGER`.

DOC's lean: **(b)** is enough. Documentation drives discoverability;
if `REACTIVE SKILL` isn't mentioned, it effectively doesn't exist
from a customer perspective. The undocumented parser alias becomes
engine-internal trivia that doesn't need a doc story.
