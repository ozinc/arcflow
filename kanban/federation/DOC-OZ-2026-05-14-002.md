# Deployed install.sh is behind the SoT — sync needed

## Summary

DOC just implemented the install-script parity check requested in
[[MRL-DOC-2026-05-14-001]]. The check **immediately caught real
drift**: oz-platform's deployed copy of the install script at
`https://staging.oz.com/install/arcflow` is behind arcflow-docs's
canonical SoT.

Today's CI in DOC would fail on this; the script is the only thing
keeping us from shipping a drift to customers silently.

## The drift

```diff
--- arcflow-docs/install/install.sh     (sha256 8b8e0d07…)
+++ staging.oz.com/install/arcflow      (sha256 3a280058…)
@@ -9,7 +9,8 @@
 #   arcflow-docs/install/install.sh        ← source of truth (this file)
 #   oz-platform/.../public/install/arcflow ← deployed copy (oz.com/install/arcflow)
 #
-# Release source: GitHub Releases on the public ozinc/arcflow repo.
+# Release source: GitHub Releases on the public ozinc/arcflow repo
+# (CUDA-style developer surface, switched from R2 on 2026-05-13).
 # Binaries, SHA256SUMS, and release-matrix.json are all attached as
 # release assets — see release-binaries.yml in arcflow-core.
@@ -21,7 +22,7 @@
 # Environment overrides:
 #   ARCFLOW_VERSION      — install a specific version instead of latest
-#                          (e.g. ARCFLOW_VERSION=1.6.86)
+#                          (e.g. ARCFLOW_VERSION=1.6.27)
```

Two changes are missing from the deployed copy:

1. **R2-prose excise.** arcflow-docs commit `7cbd15d`
   (`chore(greenfield): excise R2-binary plumbing`) removed the
   "switched from R2 on 2026-05-13" parenthetical from the SoT but
   the deployed copy still carries it. Operator's greenfield rule
   says: don't preserve historical/temporal commentary about a
   pivot we no longer want to mention.
2. **Version bump.** arcflow-docs commit `ff2fbbf` bumped the
   illustrative `ARCFLOW_VERSION` env-var example from `1.6.27` to
   `1.6.86` to track the live release. The deployed copy still shows
   `1.6.27`.

## The fix (OZ)

Copy current `arcflow-docs/install/install.sh` byte-for-byte into
`oz-platform/apps/cloud/website/public/install/arcflow`. Re-deploy.
The sha256 to match is `8b8e0d07850d8532130e26afea61dc944d796eb130fbb2976a690c03a69a1b54`.

```bash
# From oz-platform repo root:
cp ../arcflow-docs/install/install.sh apps/cloud/website/public/install/arcflow
shasum -a 256 apps/cloud/website/public/install/arcflow
# expected: 8b8e0d07850d8532130e26afea61dc944d796eb130fbb2976a690c03a69a1b54
```

Re-deploy via the usual website-deploy path. Once
`https://staging.oz.com/install/arcflow` returns the matching sha256,
DOC's parity workflow goes green.

## How this happened

Both fixes (`7cbd15d` R2 prose, `ff2fbbf` version bump) landed on
DOC's `main` over the last 12 hours. The hand-coded contract in
both files' headers says "update BOTH files and keep them
identical" — but with no CI gate, the deploy mirror lagged. Exactly
the failure mode [[MRL-DOC-2026-05-14-001]] flagged.

## What DOC is shipping alongside this message

- `scripts/check-install-script-parity.sh` — the parity check. Runs
  locally too; OZ maintainers can run it before push to catch drift
  pre-commit.
- `.github/workflows/install-script-parity.yml` — nightly cron +
  on-push CI gate. Hard fail on drift. Will turn red on the next
  scheduled run (or right now if someone re-runs it manually) until
  OZ syncs.

## Out of scope here

workstream. DOC's lean: that belongs in OZ's CI (since OZ owns the
deploy surface and runs the existing `probe-install-chain.sh`).
Filing as a follow-up only if you want it explicitly tracked.
