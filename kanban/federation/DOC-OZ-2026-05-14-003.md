# install.sh cache-control too lax — 5-min window blocks fast-propagation of fixes

## What's served today

```
$ curl -sSL --max-time 15 -I https://staging.oz.com/install/arcflow

age: 0
cache-control: public, max-age=300, must-revalidate
etag: "3b7bfd9ca73b7270f4db182d5551f6ea"
last-modified: Thu, 14 May 2026 16:24:33 GMT
```

`max-age=300` = browsers + intermediate caches MAY hold the install.sh
for 5 minutes before re-validating. `must-revalidate` is good but only
kicks in after the 5-min window expires.

## Why this matters

`install.sh` is the customer-facing entry point. A bug in install.sh —
e.g. a regression in the version-resolution logic, an asset-name mismatch
after a release rename, a security fix — needs to propagate to customers
**immediately**, not after a 5-minute cache window.

The `install-script-parity` workflow on arcflow-docs catches drift
between the SoT (`arcflow-docs/install/install.sh`) and the deployed
copy (`oz-platform/apps/cloud/website/public/install/arcflow`). It
verifies sha256 equality. **But** if a customer's browser/proxy has
cached the deployed copy for the full 5-min window, sha256 equality at
the source doesn't propagate to the customer for those 5 minutes.

Compounding: corporate proxies and ISP edge caches often respect
`max-age` more aggressively than `must-revalidate`, so the window can
stretch beyond 5 minutes in practice.

## Industry-standard pattern for install scripts

The reference shape (from the user-facing rendering of redirect-link
caching docs, which apply equally to script content):

```
Cache-Control: no-cache, no-store, must-revalidate
Pragma: no-cache
Expires: 0
```

This is what `get.docker.com`, `sh.rustup.rs`, and `install.python-poetry.org`
all return for their install scripts. Same reasoning: install scripts
must be fetch-fresh-on-every-request because:

- Security fixes need <1-minute global propagation
- Asset-URL changes (post-release renames, like the v1.6.x → v0.7.x bump
  earlier today) must take effect immediately
- Customers running `curl | sh` from corporate networks shouldn't get
  a stale cached copy from their proxy

## Suggested fix

Update the cache-control headers in oz-platform's Vercel/CDN config for
the `/install/arcflow` route:

```json
// vercel.json or equivalent
{
  "headers": [
    {
      "source": "/install/arcflow",
      "headers": [
        { "key": "Cache-Control", "value": "no-cache, no-store, must-revalidate" },
        { "key": "Pragma",        "value": "no-cache" },
        { "key": "Expires",       "value": "0" }
      ]
    }
  ]
}
```

After deploy: `curl -fsSL -I https://staging.oz.com/install/arcflow`
should show the strict headers.

## Why not max-age=0 with revalidate?

`Cache-Control: max-age=0, must-revalidate` is the minimum acceptable
behavior. `no-cache, no-store, must-revalidate` is the maximum guarantee:
no intermediate cache may even *attempt* to store the response. For a
short, hot-path script like install.sh, the request volume is low
enough that the no-cache pattern won't hurt origin load. The conservative
shape is the right call.

## Out of scope (different surface)

- **Binary assets** (`/releases/download/<tag>/<asset>`) **should** be
  long-cache (immutable per-version URL). Don't apply the strict headers
  there. GitHub already handles this correctly.
- **`/releases/latest/<asset>`** redirects should pass through to the
  versioned asset — short-cache the 302 redirect (5 min is fine here
  because the version-discovery path is the same shape), long-cache the
  underlying asset.
- The install script itself, the manifest (`release-matrix.json`), and
  `SHA256SUMS` are the three "must be fresh" surfaces. Manifest and
  SHA256SUMS are served by GitHub directly; only install.sh is the OZ
  responsibility.
