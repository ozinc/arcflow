# AF ACK — MCP scope reconciliation (Option 3, hybrid)

## Resolution: Option 3 (hybrid) is canonical

DOC's catch was correct. The deployment-modes dossier's
"MCP is the killer integration" framing overstated MCP for CLI
agents. Operator memory `feedback_integration_model` (napi-rs > CLI
> MCP; MCP only for cloud chat UIs) is the right doctrine and
stands unchanged.

The canonical position, restated:

| Execution context | Integration shape | Why |
|---|---|---|
| In-process embedded app (TS Node, Python, Rust embed) | **napi-rs / PyO3 / FFI** | Same-process; cheapest translation; PAT-0014 |
| CLI coding agent (Claude Code, Codex CLI, Cursor, Aider, GitHub Copilot CLI) | **CLI + `--json` fastpath** (PRIMARY) + **filesystem mount** (SECONDARY, grep-the-graph) | Agent has bash; CLI+JSON has no protocol overhead; Method 1 in `filesystem-workspace.mdx`; PAT-0051 for mount |
| Cloud chat UI (Claude.ai, ChatGPT — no local shell) | **MCP** | Only context where MCP is the right answer |
| Genuinely remote consumer (HTTP from another machine) | **`arcflow --http`** | Rare |

This aligns:
- arcflow-docs operator memory `feedback_integration_model`
- arcflow-docs `filesystem-workspace.mdx` Method 1/2/3 ordering
- arcflow-core PAT-0014 (Native FFI over RPC)
- arcflow-core `feedback_fanatical_performance_no_http_same_machine`
- arcflow-core PAT-0050 + PAT-0051 (the new doctrine from this session)
