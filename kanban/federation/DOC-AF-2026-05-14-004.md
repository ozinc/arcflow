# v0.7.1 release-binaries workflow failed — Windows daemon build error

## Summary

The release workflow triggered by the `v0.7.1` tag (commit `2cc359f8`)
completed: **failure** at 15m17s. **All 6 unix targets built successfully**;
both Windows targets failed at the Build step. The
`Publish to ozinc/arcflow (GitHub Releases)` job was **skipped** because the
matrix requires all builds to succeed before publishing.

Net effect: **no GitHub Release was created for v0.7.1**. `install.sh`
still resolves to v1.6.86, and the v0.7.1 convention broadcast can't be
observed via the install surface.

| `Publish to ozinc/arcflow (GitHub Releases)` | ⏭️ skipped |

## Root cause

Both Windows jobs fail with:

```
error[E0433]: cannot find `unix` in `os`
```

The Windows build command in the workflow:

```yaml
cargo build --release --bin arcflow --bin arcflow-daemon --bin arcflow-mcp --target aarch64-pc-windows-msvc
```

`arcflow-daemon` binds Unix Domain Sockets via
`std::os::unix::net::UnixListener` — not available on Windows. The
CHANGELOG entry for v1.6.87 explicitly says *"Windows daemon is not in
this release"*, but the workflow still tries to build it cross-platform.

## Suggested fixes (AF's call)

**(a) Matrix-conditional binary list** (cleanest, smallest change):
exclude `--bin arcflow-daemon` from the Windows build steps. Build only
`arcflow` + `arcflow-mcp` on Windows. Documented as intentional in the
existing CHANGELOG note.

**(b) Skip Windows targets entirely** from the v0.7.1 release until
daemon Windows support lands. Simplest but reduces the matrix.

**(c) Add `#[cfg(unix)]` gates to `arcflow-daemon`'s UDS code** so the
crate compiles on Windows (no-op build that doesn't bind UDS). Most
invasive; usable as a precursor to actual Windows-named-pipe support.

DOC's lean: **(a)**. Matches the CHANGELOG documentation; preserves
matrix coverage for `arcflow` and `arcflow-mcp` on Windows; one-line YAML
change.
