# Rust SDK · Temporal + Spatial Parquet Ingestion

The same three working steps as the Python version (`../`) — Parquet
load, windowed temporal queries, cross-engine validation — routed
through the shipped Rust SDK + the Rust `parquet` / `arrow` / `duckdb`
crates. The whole pipeline runs in one Rust process: one read of the
columnar source feeds both ArcFlow's graph and a DuckDB in-memory
table; the validation loop asks the same questions of both engines.

## When the Rust version pays off

- **Ingest pipelines that are Rust services.** When the Parquet
  consumer is itself a Rust binary (microservice, control plane,
  realtime relay), the same process can drive ingest + query +
  validation without a Python shim.
- **Cross-engine validation as a CI gate.** The validate binary
  exits non-zero on any disagreement between ArcFlow and DuckDB on
  the same Parquet input — drop it into CI as a structural correctness
  check.
- **In-process bulk-load performance.** No FFI marshalling tax; the
  `bulk_create_*` path lands at ~1 M ops/sec for the large-batch case.

## Run

```bash
cargo run --release --bin 02-parquet-load
cargo run --release --bin 04-temporal-queries
cargo run --release --bin 05-validate-vs-duckdb
```

The Parquet fixture (`../data/sample.parquet`) is generated by the
Python `00-make-sample.py` — run that once from the cookbook root
before the Rust binaries.

## Cargo layout

```
rust/
├── Cargo.toml                          # arcflow-sdk, parquet, arrow, duckdb (bundled)
├── src/
│   ├── lib.rs                          # Parquet → BOTH ArcFlow and DuckDB
│   └── bin/
│       ├── 02-parquet-load.rs          # bulk-load + sanity counts
│       ├── 04-temporal-queries.rs      # windowed per-entity queries
│       └── 05-validate-vs-duckdb.rs    # cross-engine assertion gate
```
